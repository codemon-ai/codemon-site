## 1. 핵심 컨셉

**"사람은 의사결정만, AI가 실행을 전담하는 외주 개발"**

클라이언트가 요구사항을 말하면, AI 에이전트 팀이 PRD 작성 → 스펙 정의 → 개발 → QA → 납품까지 전 과정을 수행한다. 클라이언트는 전용 사이트에서 모든 진행 상황을 실시간으로 확인하고, 주요 단계마다 승인/피드백을 제공한다.

---

## 2. 워크플로우

```
[Phase 1: 인터뷰 & 요구사항]
클라이언트 인터뷰
    ↓
PRD(요구사항 정의서) 자동 생성
    ↓
클라이언트 리뷰 & 수정 요청
    ↓
PRD 확정 ✅ (클라이언트 승인)

[Phase 2: 스펙 & 설계]
PRD → 기술 스펙 문서 자동 변환
    ↓
아키텍처 설계, 기술 스택 제안
    ↓
WBS(작업 분해), 일정 산출
    ↓
견적서 자동 생성
    ↓
클라이언트 승인 ✅ → 계약

[Phase 3: 개발]
GitHub 레포 자동 생성
    ↓
이슈 자동 분해 (PRD → GitHub Issues)
    ↓
Development Agent가 이슈별 개발
    ↓
PR 생성 → QA Agent 자동 코드 리뷰
    ↓
PM Agent 진행률 관리 & 클라이언트 리포트

[Phase 4: 검수]
스테이징 환경 자동 배포
    ↓
클라이언트 검수 시작
    ↓
QA 체크리스트 자동 생성
    ↓
버그/수정 → GitHub Issue 자동 등록
    ↓
수정 → 재검수 (반복)
    ↓
QA 전체 통과 ✅

[Phase 5: 납품]
프로덕션 배포
    ↓
인수인계 문서 자동 생성
    ↓
클라이언트 최종 확인 ✅
    ↓
잔금 수령
```

## 3. AI 에이전트 구성

### PM Agent (프로젝트 매니저)
- **역할**: 전체 프로젝트 관리, 일정 추적, 클라이언트 커뮤니케이션
- **담당 업무**:
  - 인터뷰 내용 → PRD 자동 생성
  - WBS 분해, 마일스톤 설정
  - 진행률 대시보드 업데이트
  - 클라이언트에게 주간/일간 리포트
  - 이슈 우선순위 조정
  - 일정 지연 시 알림 & 대안 제시

### Development Agent (개발자)
- **역할**: 실제 코드 작성, PR 생성
- **담당 업무**:
  - GitHub Issue → 브랜치 생성 → 코드 작성
  - PR 생성 (자동 테스트 포함)
  - 코드 리뷰 피드백 반영
  - 기술 문서 자동 작성
  - 스테이징 배포

### QA Agent (품질 관리)
- **역할**: 코드 리뷰, 테스트, 검수 관리
- **담당 업무**:
  - PR 자동 코드 리뷰
  - 테스트 케이스 자동 생성 & 실행
  - QA 체크리스트 생성 & 관리
  - 클라이언트 검수 피드백 → Issue 변환
  - 버그 재현 & 분류

### 주요 기능
| 기능 | 설명 |
|------|------|
| 실시간 진행률 | GitHub Issues 기반 자동 계산 |
| 문서 열람 | PRD, 스펙, 아키텍처 문서 실시간 확인 |
| 승인 워크플로우 | PRD 승인, 스펙 승인, 검수 승인 버튼 |
| 이슈 트래커 | GitHub Issues 연동, 클라이언트가 직접 버그 등록 가능 |
| QA 체크리스트 | 검수 항목별 통과/실패 실시간 표시 |

| 타임라인 | 모든 활동 로그 (누가, 언제, 무엇을) |
| 결제 연동 | 마일스톤별 자동 청구 |

## 5. GitHub 이슈 관리 체계

### 이슈 라벨
```
phase/requirement    — 요구사항 단계
phase/spec           — 스펙 단계
phase/development    — 개발 단계
phase/qa             — QA 단계

type/feature         — 신규 기능
type/bug             — 버그
type/enhancement     — 개선
type/documentation   — 문서

priority/critical    — 긴급
priority/high        — 높음
priority/medium      — 보통
priority/low         — 낮음

agent/pm             — PM Agent 담당
agent/dev            — Dev Agent 담당
agent/qa             — QA Agent 담당

status/client-review — 클라이언트 검토 대기
status/approved      — 승인됨
status/in-progress   — 진행중
status/blocked       — 차단됨
```

### 이슈 자동 생성 흐름
```
PRD 확정
  ↓
PM Agent가 PRD를 분석
  ↓
Epic Issue 생성 (기능 단위)
  ↓
Sub-Issue 자동 분해 (작업 단위)
  ↓
Dev Agent에 할당
  ↓
각 이슈에 브랜치 자동 생성
```

## 6. 기술 스택 (예상)

### 포털 사이트
- **프론트엔드**: Next.js + Tailwind
- **백엔드**: FastAPI 또는 Next.js API Routes
- **DB**: PostgreSQL (Supabase)
- **인증**: OAuth (Google, GitHub)
- **실시간**: WebSocket (진행률 업데이트)

### AI 에이전트
- **프레임워크**: OpenClaw 기반
- **코딩 에이전트**: Claude Code (YOLO 모드)
- **이슈 관리**: GitHub API (gh CLI)
- **배포**: Vercel (스테이징 + 프로덕션)

### 인프라
- **CI/CD**: GitHub Actions
- **스테이징**: Vercel Preview Deployments
- **모니터링**: 포털 대시보드 내장

## 7. 수익 모델

| 모델 | 설명 |
|------|------|
| 프로젝트 단가 | 기존 외주 대비 50~70% 가격 (AI가 실행하니까) |
| 마일스톤 결제 | 계약금(30%) → 중간(30%) → 잔금(40%) |
| 유지보수 구독 | 납품 후 월정액 유지보수 |
| 긴급 대응 | 추가 요금 (24시간 내 핫픽스) |

### 가격 경쟁력
- **기존 외주**: 개발자 인건비 기반 → 비쌈
- **AI 외주**: API 비용 + 플랫폼 수수료 → 저렴
- **투명성**: 클라이언트가 진행상황을 보니까 신뢰도 ↑

## 8. 차별점

| 기존 외주 | AI 전자동 외주 |
|----------|---------------|
| 블랙박스 (뭘 하는지 모름) | 전 과정 투명 공개 |
| PM 한 명이 여러 프로젝트 | AI PM이 프로젝트 전담 |
| 소통 비용 큼 (회의, 메일) | 포털에서 비동기 소통 |
| QA는 최종 단계에서만 | PR마다 자동 코드 리뷰 |
| 일정 지연 빈번 | 실시간 일정 추적, 자동 알림 |
| 산출물 인수인계 어려움 | 문서 자동 생성, GitHub 레포 그대로 전달 |

## 9. 리스크 & 대응

| 리스크 | 대응 |
|--------|------|
| AI 코드 품질 불안 | QA Agent 3중 검증 (코드리뷰 + 자동테스트 + 수동검수) |
| 복잡한 요구사항 이해 실패 | 인터뷰 단계에서 사람이 개입, PRD 승인 필수 |
| 클라이언트 신뢰 부족 | 포털 투명성으로 해소, 첫 프로젝트 할인 |
| 보안/NDA | 프라이빗 레포, 접근 권한 관리 |
| AI API 비용 급증 | 프로젝트별 토큰 사용량 모니터링, 상한선 설정 |
