# Phase 0-4: 패키지 상품 설계 초안

> **작성일**: 2026-02-25
> **목적**: COO 요구 "미리 만들어놓은 복합 패키지 상품" 설계

---

## 1. 요구사항

### COO 요구
- 현재: 수기 견적으로 복합상품 프로모션 처리
- 원하는 것: 미리 패키지 상품을 만들어놓고 판매
- 예: "방콕 3박4일 패키지" = 호텔 3박 + 공항픽업 + 시티투어 + 스파

### 비즈니스 시나리오

```
관리자가 패키지 상품 등록
  ├── 호텔: Marriott 3박 (필수)
  ├── 차량: 공항 픽업 (필수)
  ├── 투어: 시티투어 (필수)
  └── 스파: Thai Spa (선택 추가)

패키지 가격: 개별 합산 120만원 → 패키지 특가 99만원 (17% 할인)

고객이 패키지 주문
  → Order 1개 생성
  → Booking 4개 생성 (각 상품별)
  → Invoice 1개 발송 (패키지 가격 기준)
  → 각 Booking별 벤더 확정 → 바우처 발송
```

## 2. 설계 옵션

### 옵션 A: Product 확장 (PACKAGE 타입) ✅ 추천

`products` 테이블에 `productType = 'PACKAGE'`로 등록하고, 구성 상품은 별도 테이블로 관리.

```
products (type=PACKAGE)
  ├── name: "방콕 3박4일 패키지"
  ├── price: 990000 (패키지 특가)
  ├── detail: { packageType, description, ... }
  └── package_items (1:N)
       ├── childProductId: 호텔
       ├── childProductId: 차량
       ├── childProductId: 투어
       └── childProductId: 스파 (optional)
```

**장점**: 기존 상품 리스트/카테고리/검색에 자연스럽게 노출
**단점**: 재고/가격 동기화 복잡 (구성 상품 가격이 바뀌면?)

### 옵션 B: 별도 Package 도메인

**장점**: 구분 명확
**단점**: 중복 코드, 고객 UI에서 별도 처리 필요

## 3. 데이터 모델

### package_items 테이블

```typescript
export const packageItems = pgTable('package_items', {
  id: serial('id').primaryKey(),
  
  // 패키지 상품 (products.productType = 'PACKAGE')
  packageProductId: integer('package_product_id')
    .notNull()
    .references(() => products.id, { onDelete: 'cascade' }),
  
  // 구성 상품
  childProductId: integer('child_product_id')
    .notNull()
    .references(() => products.id),
  
  // 옵션
  quantity: integer('quantity').default(1),
  nights: integer('nights'),         // 호텔 박수
  isOptional: boolean('is_optional').default(false),
  // 필수(false) vs 선택 추가(true)
  
  additionalPrice: decimal('additional_price', { precision: 12, scale: 2 }),
  // 선택 상품 추가 금액 (optional일 때)
  
  sortOrder: integer('sort_order').default(0),
  memo: text('memo'), // "수페리어룸 기준" 등
  
  createdAt: timestamp('created_at').defaultNow().notNull(),
});
```

## 4. Order와의 연동

패키지 상품 주문 시:

```
1. 고객이 패키지 상품 선택
2. Order 생성 (status: REQUESTED)
3. 패키지 구성 상품별 Booking 자동 생성:
   - Booking 1: 호텔 (필수) → productId: 호텔상품, orderId: Order
   - Booking 2: 차량 (필수) → productId: 차량상품, orderId: Order
   - Booking 3: 투어 (필수) → productId: 투어상품, orderId: Order
   - Booking 4: 스파 (선택) → productId: 스파상품, orderId: Order
4. Invoice: Order 기준 1개 (패키지 특가 기준)
5. Payment: Order 기준 (계약금/잔금 가능)
6. 각 Booking 벤더 확정 → 개별 바우처 발송
```

## 5. 가격 정책

| 항목 | 설명 |
|------|------|
| 패키지 가격 | `products.price` (패키지 특가) |
| 개별 합산 가격 | 구성 상품 가격 합계 (비교 표시용) |
| 할인율 | `(합산 - 패키지가) / 합산 × 100` (자동 계산) |
| 선택 상품 | 기본가에 `additionalPrice` 추가 |
| 매입가 | 각 구성 상품의 costPrice 합산 |

### 가격 동기화 문제

구성 상품 가격이 바뀌면 패키지 가격은?
- **권장**: 패키지 가격은 독립 관리 (수동 설정)
- 구성 상품 가격 변경 시 관리자에게 알림만 (자동 변경 ❌)

## 6. 어드민 UI

### 패키지 상품 등록 폼

```
기본 정보 (기존 상품 폼과 동일)
  └── 상품분류: 패키지 전용 카테고리
  └── 가격: 패키지 특가

패키지 구성 (신규 섹션)
  ├── [+ 상품 추가] 버튼 → 상품 검색 모달
  ├── 드래그 정렬
  └── 각 항목:
       - 상품명 (자동)
       - 수량 / 박수
       - 필수/선택
       - 추가 금액 (선택일 때)
       - 메모

유효 기간
  └── 판매 시작일 ~ 종료일
```

## 7. COO 추가 확인 필요

1. 패키지 구성 상품을 **고객이 변경**할 수 있는지? (호텔 업그레이드 등)
2. 패키지 내 **일부 상품만 취소** 가능한지?
3. 패키지 **날짜가 상품마다 다를** 수 있는지? (호텔 3박 + 2일차 투어 + 3일차 골프)
4. 같은 구성으로 **날짜만 다른 패키지**를 쉽게 만들 수 있어야 하는지?
5. 패키지 **재고 관리**: 패키지 자체 수량 제한 vs 구성 상품 재고 기반
