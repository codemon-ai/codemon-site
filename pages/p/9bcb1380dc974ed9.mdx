# BlueMango v2 — Admin PRD

> 태국 여행 예약 플랫폼 어드민 사이트 재구축 PRD
> 기준일: 2026-02-23

---

## 1. 개요

### 1.1 배경
BlueMango v1 Admin은 **Refine + Ant Design** 기반으로 개발되었으나 다음 문제가 있어 전면 재구축한다:
- Refine + Ant Design 프레임워크 과중 — 에러 빈번, 번들 사이즈 과대
- 카테고리별 별도 페이지/컨트롤러 (NestJS 21개 컨트롤러, 230개 엔드포인트)
- LEGACY 코드 부채 — 무통장입금 독립, 이중 상태, Order 잔재 등

### 1.2 목표
- **shadcn/ui 기반** 모던 어드민 UI 전면 재구축
- 카테고리별 분리 → **통합 상품 관리** (타입/카테고리 필터)
- LEGACY 문제점 전면 해결 (무통장입금 통합, 이중 상태 제거 등)
- **1인 개발자(+AI) 유지보수** 최적화
- 라우트 수 80+ → 35개로 대폭 축소

### 1.3 기술 스택

| 영역 | 기술 | 선택 이유 |
|------|------|-----------|
| 프레임워크 | Next.js 15 (App Router) | 서버리스, Vercel 네이티브 |
| Backend | Next.js API Routes + Server Actions | NestJS 제거 |
| DB | Supabase PostgreSQL (Seoul) | 통합 인프라 |
| ORM | Drizzle ORM | 35KB, 서버리스 최적 |
| Auth | Better Auth (Drizzle 어댑터) | 카카오/구글/네이버 지원 |
| Storage | Supabase Storage | S3 대체, CDN |
| UI | shadcn/ui + Tailwind CSS | 경량, 프리미엄 DS |
| 에디터 | Tiptap | 리치텍스트 (상품 설명, 콘텐츠) |
| 차트 | Recharts | 대시보드 통계 |
| 유효성검사 | Zod | 전 레이어 통일 |
| 테스트 | Playwright | E2E |
| 배포 | Vercel | 서버리스, 자동 스케일링 |

## 2. 사용자 역할 및 권한 (RBAC)

### 2.1 역할 계층
```
SUPER_ADMIN > MANAGER > ADMIN > STAFF > SUPPLIER
```
상위 역할은 하위 역할의 모든 권한 포함.

| 역할 | 설명 |
|------|------|
| SUPER_ADMIN | 최고 관리자 (모든 기능 + 시스템 설정) |
| MANAGER | 매니저 (ADMIN + 직원 관리 + 약관/설정) |
| ADMIN | 관리자 (대부분의 어드민 기능) |
| STAFF | 직원 (조회 + 제한된 수정) |
| SUPPLIER | 공급처 (자기 상품/정산만) |

### 2.2 인증
- 이메일/비밀번호 로그인 (bcrypt salt 10)
- JWT: Access Token 1시간 / Refresh Token 1일
- 관리자 활동 로그: 모든 작업 기록 (action, resource, resourceId, data, ipAddress, userAgent)

## 5. 도메인별 어드민 기능 상세

### 5.1 대시보드 (/)
- 매출 요약 (일/주/월) — Recharts
- 예약 현황 (상태별 카운트)
- 인기 상품 (예약 수/매출 기준)
- 최근 예약 목록
- 사용자 통계

### 5.2 예약관리

#### 예약 목록 (/bookings)
**필터 체계** (v1의 별도 메뉴 대체):
- 상태: PENDING | INVOICE_SENT | PAYMENT_COMPLETED | VOUCHER_SENT | COMPLETED | CANCELLED
- 상품타입: ALL | HOTEL | GOLF | VEHICLE | TOUR | ETC
- 결제방법: ALL | CREDIT_CARD | BANK_TRANSFER | VIRTUAL_ACCOUNT ← 무통장입금 필터
- 기간: 예약일 / 이용일 range
- 검색: 예약번호 / 고객명 / 상품명
- 정렬: 최신순 / 이용일순 / 금액순

#### 예약 상세 (/bookings/[id])
- 예약 기본 정보 (bookingReference, 상태, 날짜, 고객 정보)
- 상태 변경 (상태 머신 규칙 적용)
- 인보이스 탭: 생성/발송/PDF 생성
- 바우처 탭: 생성/발송/PDF 생성
- 결제 내역
- 상태 변경 이력 (BookingStatusHistory)

#### 예약 상태 머신
```
PENDING → INVOICE_SENT → PAYMENT_COMPLETED → VOUCHER_SENT → COMPLETED
   │                           │                    │
   │         CANCEL_REQUESTED ←┘                    │
   │                │                               │
   └────────────────┴─────────→ CANCELLED ←─────────┘
```

| 상태 | 의미 | 트리거 |
|------|------|--------|
| PENDING | 예약 접수 | 예약 생성 시 |
| INVOICE_SENT | 인보이스 발송 | 관리자 또는 자동(instantInvoice) |
| PAYMENT_COMPLETED | 결제 완료 | PG 콜백 |
| VOUCHER_SENT | 바우처 발송 (확정) | 관리자 발송 |
| COMPLETED | 서비스 완료 | 이용일 종료 |
| CANCEL_REQUESTED | 취소 요청 | 사용자 요청 |
| CANCELLED | 취소 완료 | 관리자 승인 |

- 모든 상태 변경은 BookingStatusHistory에 기록 (누가, 언제, 왜)
- CANCELLED는 terminal (변경 불가)
- requiresConsultation=true: PENDING에서 관리자 확인 필수
- instantInvoice=true: 예약 즉시 INVOICE_SENT 자동 전이

#### 인보이스
- invoiceNumber: INV\{timestamp\}\{random3digits\}
- 자동 생성: issuedDate, dueDate(+7일), taxAmount(10%), totalAmount
- 공개 링크: /public/invoice/[uuid] (로그인 불필요)
- PDF 생성: React Email + @react-pdf/renderer
- 알림: 알림톡 + Email

#### 바우처
- voucherNumber: VCH\{timestamp\}\{random3digits\}
- 전제조건: status = PAYMENT_COMPLETED
- 상태: ISSUED → SENT → USED / EXPIRED / CANCELLED
- 공개 링크: /public/voucher/[uuid]
- PDF 생성 + 알림 발송

#### 예약 생성 (/bookings/create)
- 관리자가 직접 예약 생성 (개인결제 상품 등)
- 필수: productId, quantity, startDate, endDate, guestName, guestEmail, guestPhone

#### 결제 관리 (/payments) 🆕
- Payment 엔티티 중심 관리
- 결제 수단별 필터: CREDIT_CARD | BANK_TRANSFER | VIRTUAL_ACCOUNT
- **무통장입금 관리** (v1의 /bank-transfers 대체):
  - method=BANK_TRANSFER, status=PENDING인 결제 필터
  - 관리자 입금 확인: PENDING → COMPLETED
  - 입금 카테고리: PaymentCategory 테이블 (카카오뱅크, 방콕은행 등)
  - 통화: KRW 또는 THB
- 환불 처리: 전액/부분 환불

### 5.3 상품관리

#### 상품 목록 (/products) 🆕
- **통합 목록** (v1의 카테고리별 분리 대체)
- 필터: 상품타입(HOTEL/GOLF/VEHICLE/TOUR/ETC), 카테고리, 상태(DRAFT/ACTIVE/INACTIVE)
- 검색: 상품명, 상품코드

#### 상품 생성/수정 — 타입별 동적 폼
```
상품 폼
│
├── [공통 필드] ─── 항상 노출
│   name, nameEn, description, category, price, costPrice,
│   discountPrice, status, supplier, images, tags, memo
│
├── HOTEL ─── 호텔 전용
│   rooms[], checkInTime, checkOutTime, amenities[], facilities[]
│   roomAvailability (날짜별 가격/잔여)
│
├── GOLF ─── 골프 전용
│   holes, courseType, greenFee, caddyFee, cartFee, availableTimeSlots[]
│
├── VEHICLE ─── 차량 전용 (v2 신규 엔티티)
│   vehicleType, seatingCapacity, driverIncluded, fuelType
│   pickupLocations[], dropoffLocations[], pricePerHour, pricePerDay
│
├── TOUR ─── 투어 전용 (v2 신규 엔티티)
│   tourType(private/land/ocean), duration, maxParticipants
│   itinerary[], inclusions[], exclusions[], meetingPoint, meetingTime
│
└── ETC ─── 기타 전용
    etcType(spa/restaurant/show/fast-track/visa), operatingHours, duration
```

#### 상품 공통 필드
- 식별: code(고유, 자동생성), name(KR), nameEn(EN)
- 설명: description (Tiptap 리치텍스트)
- 분류: productType, categoryId(FK, 필수)
- 가격: price(판매가), costPrice(매입가), discountPrice(할인가)
- 위치: address, latitude, longitude, regionCode
- 상태: status (DRAFT / ACTIVE / INACTIVE)
- 이미지: ProductImage 엔티티 (1:N) — imageType(MAIN/ROOM/FACILITY/MENU/OTHER)
- 예약설정: maxQuantity, minQuantity, requiresConsultation, instantInvoice
- 공급처: supplierId (FK, 필수)

### 5.6 정산관리 🆕

#### 정산 목록 (/settlements)
- 공급처(Supplier)별 월 단위 정산
- settlementNumber: ST\{YY\}\{MM\}\{supplierID\}
- 정산 계산:
  ```
  commissionAmount = bookingPrice × supplier.commissionRate / 100
  settlementAmount = bookingPrice - commissionAmount
  ```
- 상태: PENDING → CONFIRMED → PAID (또는 CANCELLED)

#### 공급처 관리 (/suppliers)
- CRUD: name, businessNumber, contactPerson, phone, email
- 은행: bankName, bankAccount, accountHolder
- 정산: commissionRate(%), settlementCycle, settlementDay
- 계약: 시작일, 종료일, 계약서 URL
- 상태: active / inactive / suspended

## 6. 알림 시스템 (어드민 트리거)

| 이벤트 | 알림톡 | Email | 어드민 트리거 |
