# AI 스킬 고수가 되는 법 — description부터 체이닝까지

> 시리즈: AI한테 일 가르치기 (3/3)

## 여기까지 온 당신

1편에서 스킬 개념을 알았고, 2편에서 자동화를 걸었다.
이제 남은 건 **"잘"** 만드는 것이다.

같은 기능인데 결과 품질이 천차만별인 이유?
**description 차이**다. 그리고 **실패에 대비하는 설계** 차이다.

## Description 작성법: 프로 vs 아마추어

### 아마추어

```
이름: summarizer
설명: 요약해주는 스킬
```

이러면 AI가 뭘 요약하라는 건지, 어떤 형식으로 해야 하는지, 언제 이 스킬을 써야 하는지 모른다. 매번 결과가 다르다.

<!-- SCREENSHOT-1: 아마추어 description으로 만든 스킬의 들쭉날쭉한 결과 2개 나란히 비교 (같은 입력인데 다른 출력) -->

### 프로

```
이름: youtube-summarizer
설명: YouTube 영상 URL을 받으면 자막을 기반으로 내용을 요약한다.
      출력 형식:
      - 제목 (한 줄)
      - 핵심 포인트 3~5개 (글머리 기호)
      - 한 줄 감상 ("이 영상은 ~한 사람에게 추천")
      - 원본 URL
      트리거: "이거 요약해줘", "영상 정리", YouTube URL 공유 시 자동.
      주의: 영상이 30분 이상이면 챕터별로 나눠서 요약.
```

<!-- SCREENSHOT-2: 프로 description으로 만든 스킬의 일관된 결과 — YouTube URL 넣고 깔끔하게 정리된 요약 출력 -->

### 프로 description의 공식

```
[언제 발동하는지] + [뭘 하는지] + [출력 형식] + [예외 처리]
```

이 4가지만 넣으면 된다. 하나씩 보자:

**1. 언제 발동하는지 (트리거)**
```
"회의록 정리해줘", "미팅 노트" 하면 실행.
회의 내용이 텍스트로 입력되면 자동 인식.
```

**2. 뭘 하는지 (액션)**
```
입력받은 회의 내용을 구조화된 회의록으로 변환한다.
```

**3. 출력 형식 (포맷)**
```
- 날짜/참석자
- 논의 사항 (번호)
- 결정 사항
- 액션 아이템 (담당자 + 기한)
```

**4. 예외 처리 (엣지 케이스)**
```
참석자가 명시되지 않으면 "참석자: 미기재"로 표시.
액션 아이템에 기한이 없으면 "기한: 미정"으로 표시.
```

## 실패에 대비하는 설계

AI도 실수한다. 중요한 건 **실패했을 때 어떻게 할지** 미리 정해두는 것.

### 원칙 1: 부분 성공도 결과를 보여줘라

```
❌ "요약 실패" → 끝
✅ "3개 중 2개만 요약 성공. 나머지 1개는 접근 불가."
```

### 원칙 2: 되돌릴 수 있게 만들어라

파일 이동 스킬이라면:
```
이동 전에 원본 경로를 기록해둔다.
"되돌려줘" 하면 원래 위치로 복구한다.
```

### 원칙 3: 모르면 물어보게 해라

```
입력이 애매하면 "회의 날짜가 언제인가요?" 같이 되물어본다.
추측해서 채우지 않는다.
```

이 3가지를 description에 넣어두면, 스킬의 **신뢰도**가 확 올라간다.

## 개발자를 위한 보너스: Claude Code 스킬

여기서부터는 개발자 영역이다.
코딩을 할 줄 안다면, Claude Code에서 **훨씬 강력한 스킬**을 만들 수 있다.

### Claude Code 스킬이 다른 점

| | Cowork 스킬 | Claude Code 스킬 |
|---|---|---|
| 대상 | 일반 사용자 | 개발자 |
| 작성 | 자연어 설명 | 마크다운 파일 (SKILL.md) |
| 실행 | Claude Desktop | 터미널 (CLI) |
| 도구 | 제한적 | 파일 시스템, Git, API, 터미널 전체 |
| 자동화 | 스케줄 태스크 | cron + 파이프라인 |

<!-- SCREENSHOT-3: Cowork vs Claude Code 비교 — 왼쪽: Claude Desktop Cowork UI / 오른쪽: 터미널에서 Claude Code 실행 화면 -->

### SKILL.md 구조

```yaml
---
name: blog-publish
description: "블로그 드래프트를 검토하고 발행한다.
  '블로그 발행해줘' 하면 실행.
  Keywords: 블로그, 발행, publish, deploy."
allowed-tools:
  - "Read"
  - "Write"
  - "Bash"
---

# blog-publish

## 워크플로우
1. drafts/ 폴더에서 대상 파일 확인
2. 마크다운 검증 (frontmatter, 이미지 경로)
3. 코드몬 승인 요청
4. 승인 시 → MDX 변환 → Git push → Vercel 배포
5. 발행 완료 알림
```

<!-- SCREENSHOT-4: 실제 SKILL.md 파일을 에디터(VS Code or 옵시디언)에서 열어둔 화면 -->

### 실전: 코드몬 팀이 쓰는 스킬들

**1. blog-publish** — 드래프트 → 검토 → 이미지 생성 → 발행 자동화
- 글 하나 발행하는 데 원래 40분 → 지금 10분

**2. docsearch** — 메모리 + 옵시디언 + iCloud 통합 검색
- "이거 문서 어디 있어?" 한 마디에 전체 검색

**3. obsidian-markdown** — 옵시디언 전용 마크다운 생성
- wikilinks, callouts, frontmatter 자동 처리

**4. project-lifecycle** — 프로젝트 생성 → 진행 → 완료 → 아카이브
- PARA 구조에 맞춰 자동으로 폴더 이동

이런 스킬들이 **파일 시스템을 공유 메모리로** 쓰면서 서로 연결된다.
A 스킬의 결과물이 B 스킬의 입력이 되는 구조. 이게 **체이닝**이다.

## 스킬 체이닝: 시스템으로 만들기

단일 스킬 → 자동화 → 체이닝. 이 3단계가 스킬의 성장 경로다.

```
Level 1: 단일 스킬
  "회의록 정리해줘" → 정리된 회의록

Level 2: 자동화
  매일 08:00 → 아침 브리핑 자동 생성

Level 3: 체이닝
  아침 브리핑 → 오늘 회의 감지 → 회의록 템플릿 준비
  → 회의 후 정리 → 액션 아이템 추출 → 주간 리포트 반영
```

<!-- SCREENSHOT-6: 위 체이닝 플로우를 시각화한 다이어그램 (Excalidraw or 간단 플로우차트 이미지 제작) -->

일반 사용자도 Level 2까지는 충분히 할 수 있다.
Level 3은 개발자가 Claude Code로 구축하면 더 강력하지만,
Cowork에서도 스킬끼리 "다음에 이 스킬 실행해줘" 식으로 연결 가능하다.

## 나만의 AI 시스템 완성하기: 체크리스트

### 초보 (1~2주)
- [ ] 스킬 1개 만들어서 매일 사용
- [ ] description 3번 이상 다듬기
- [ ] "이건 좀 다르게 해줘" 피드백으로 개선

### 중급 (3~4주)
- [ ] 스케줄 태스크 1개 설정
- [ ] 스킬 3개 이상 운용
- [ ] 파일 읽기/쓰기 스킬 1개

### 고급 (1개월+)
- [ ] 스킬 체이닝 구성
- [ ] 실패 대응 로직 포함
- [ ] 나만의 워크플로우 완성

급하게 10개 만들지 말고, **하나를 제대로** 만들어서 2주 써보자.
그 하나가 습관이 되면 두 번째는 자연스럽게 만들게 된다.
