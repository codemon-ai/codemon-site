# Order(주문) 도입 영향 분석

> 작성일: 2026-02-27 | 대상: COO, PM | 문서 유형: PRD

## 1. 한 줄 요약

Order(주문) 객체를 도입하여 "1주문 = N상품 = 1결제"를 가능하게 합니다. 동시에 PG사를 토스페이먼츠에서 KG이니시스로 전환합니다.

## 2. 배경 — 왜 바꿔야 하는가

### 현재 문제점
- 고객이 3개 상품을 예약하면 3번 결제해야 함
- 인보이스가 예약별로 개별 발행 → 관리 복잡
- 취소/환불도 건건이 처리

### 변경 목표
- 1개 주문에 여러 상품을 담아 1번에 결제
- 인보이스/바우처도 주문 단위 통합 관리
- 부분취소(일부 상품만) 지원

## 3. 현재 vs 변경 비교

| 단계 | 현재 (AS-IS) | 변경 후 (TO-BE) |
|------|-------------|----------------|
| 상품 선택 | 1개씩 예약 | 장바구니에 담아 한번에 주문 |
| 인보이스 | 예약별 개별 발행 | 주문 단위 통합 발행 |
| 결제 | 예약별 개별 결제 | 주문 단위 1회 결제 (계약금/잔금 분리 가능) |
| 바우처 | 예약별 발행 | 예약별 발행 유지 (상품마다 이용일이 다를 수 있음) |
| 취소/환불 | 전체 취소만 | 주문 내 개별 상품 취소 가능 |
| 정산 | 예약 기준 | 주문 기준 (공급사별 분배) |

## 4. 관리자 화면 변경

### 4.1 예약 목록
- 주문번호(Order Number) 컬럼 추가
- 같은 주문의 예약들을 그룹으로 묶어 표시
- 주문번호로 검색 가능

### 4.2 인보이스 관리
- 주문 단위로 통합 인보이스 발행
- 개별 예약이 아닌 주문에 연결

### 4.3 주문 관리 (신규 화면)
- 주문 목록: 주문번호, 고객명, 상품 수, 총액, 상태
- 주문 상세: 포함 예약 목록, 결제 내역, 변경 이력

### 4.4 대시보드
- KPI 기준: 예약 건수 → 주문 건수 + 예약 건수 병행
- 매출: 주문 기준 집계

## 5. 데이터 구조 변경 (비기술 설명)

### 새로 만드는 것
| 항목 | 설명 |
|------|------|
| **주문(Order)** 테이블 | 고객의 주문 정보 (주문번호, 고객, 총액, 상태) |
| **변경요청(Change Request)** 테이블 | 부분 취소, 날짜 변경 등 변경 이력 |

### 기존에서 바뀌는 것
| 항목 | 변경 내용 |
|------|----------|
| **예약(Booking)** | 주문번호(orderId) 추가 — 어떤 주문에 속하는지 연결 |
| **결제(Payment)** | 예약이 아닌 주문에 연결 + PG사 구분(토스/이니시스) 추가 |
| **인보이스(Invoice)** | 예약이 아닌 주문에 연결 |
| **장바구니(Cart)** | 주문 전환 로직 추가 |
| **정산(Settlement)** | 상태 5단계로 확장 |

## 6. 운영 영향도

| 업무 | 현재 | 변경 후 | 영향도 |
|------|------|---------|--------|
| 예약 접수 | 건별 확인 | 주문 단위 확인 → 개별 예약 처리 | 중간 |
| 인보이스 발행 | 예약별 수동 발행 | 주문 자동 통합 발행 | 개선 |
| 결제 확인 | 예약별 확인 | 주문 단위 확인 (1회) | 개선 |
| 바우처 발행 | 예약별 발행 | 예약별 발행 (변동 없음) | 없음 |
| 취소/환불 | 전체 취소 | 부분 취소 가능 → 변경요청 관리 추가 | 중간 |
| 공급사 정산 | 예약별 수동 | 주문 내 공급사별 자동 분배 | 개선 |
| CS 대응 | 예약번호로 검색 | 주문번호 + 예약번호 둘 다 검색 | 중간 |
| 매출 리포팅 | 예약 기준 | 주문 기준 집계 | 중간 |

## 7. 기존 데이터 처리

기존 예약 데이터는 자동으로 마이그레이션됩니다:
- 기존 예약 1건 = 주문 1건으로 자동 생성 (1:1 매핑)
- 기존 결제/인보이스도 새 주문에 연결
- **고객에게 보이는 변화 없음** — 기존 예약번호 유지
- 마이그레이션은 서비스 중단 없이 진행 가능

## 9. PG사 변경 안내

### 변경 내용
- **현재**: 토스페이먼츠 (Toss Payments)
- **변경**: KG이니시스 (INICIS)
- 계약 완료, 테스트 계정 수령 완료 (MID: bluemango1)

### 변경 이유
- KG이니시스가 태국 여행업 결제에 더 유리한 조건 (수수료, 정산 주기)
- 카드사 심사 진행 중

### 고객 경험 변화
- 결제 화면 디자인이 약간 다를 수 있으나 **결제 과정 자체는 동일**
- 지원 결제 수단: 카드 결제 (동일)
- 영수증/결제 내역: 이니시스 브랜드로 표시

## 12. 운영 상태 정의(POL-GEN-000) 매핑

v2 시스템은 COO가 정의한 POL-GEN-000 운영객체·상태정의를 기반으로 설계되었습니다. 다만 1단계(전액결제)에서는 일부 상태를 압축하여 운영 복잡도를 낮췄습니다. 아래는 관련 5개 객체의 매핑표입니다.

### 12.1 Order 상태 매핑

| POL-GEN-000 (운영 정의) | v2 시스템 (DB) | 설명 |
|------------------------|---------------|------|
| Requested | `PENDING` | 주문 접수 |
| Pending_check | `PENDING` | 검증은 시스템 자동 (별도 상태 불필요) |
| Payment_requested | `PENDING` | 인보이스 발송으로 결제 요청 (PENDING 유지) |
| Payment_received | `PAID` | 결제 확인 |
| Vendor_requested | `PROCESSING` | 벤더 확인 진행 |
| Confirmed | `PROCESSING` | 벤더 확정 (PROCESSING 내부 구분) |
| Voucher_sent | `PROCESSING` | 바우처 발송 완료 (Booking 쪽에서 VOUCHER_SENT) |
| In_service | `PROCESSING` | 서비스 진행 중 |
| Completed | `COMPLETED` | 종료 |
| Cancel_requested | `CANCEL_REQUESTED` | 취소 요청 |
| Cancelled | `CANCELLED` | 취소 완료 |

**핵심 차이점:**
- POL-GEN-000의 11단계 중 Pending_check / Payment_requested / Vendor_requested / Confirmed / Voucher_sent / In_service → v2에서는 **PENDING / PROCESSING으로 압축**
- 이유: D-08 결정에 따라 1단계(전액결제)에서는 세분화 불필요
- **향후 확장 경로**: 계약금/잔금 분리 결제 도입 시 DEPOSIT_PAID, BALANCE_PAID 추가 가능

### 12.2 Payment 상태 매핑

| POL-GEN-000 | v2 시스템 | 설명 |
|-------------|----------|------|
| Pending | `PENDING` | 1:1 |
| Deposit_paid | — | 1단계 미구현 (D-04: 전액결제만) |
| Paid | `COMPLETED` | v2에서는 COMPLETED로 표현 |
| Failed | `FAILED` | 1:1 |
| Refunded | `REFUNDED` | 1:1, + `PARTIAL_REFUNDED` 추가 |

### 12.3 Change_Request 상태 매핑

| POL-GEN-000 | v2 시스템 | 설명 |
|-------------|----------|------|
| Requested | `REQUESTED` | 1:1 |
| Reviewed | `REVIEWED` | 1:1 |
| Approved | `APPROVED` | 1:1 |
| Rejected | `REJECTED` | 1:1 |
| Expired | `EXPIRED` | 1:1 |

→ Change_Request는 POL-GEN-000과 **완전 일치**

### 12.4 Settlement 상태 매핑

| POL-GEN-000 | v2 시스템 (D-11 추천) | 설명 |
|-------------|---------------------|------|
| Open | `PENDING` | 네이밍만 다름 |
| Calculated | `CALCULATED` | 1:1 |
| Approved | `APPROVED` | 1:1 |
| Paid | `PAID` | 1:1 |
| Closed | — | v2는 DISPUTED 추가, Closed 대신 PAID에서 종료 |

### 12.5 Product 상태 매핑

| POL-GEN-000 | v2 시스템 | 설명 |
|-------------|----------|------|
| Draft | `DRAFT` | 1:1 |
| Reviewed | — | v2에서는 Admin 검토가 별도 상태 없음 |
| Approved | — | Draft → Active 직행 |
| Sellable | `ACTIVE` | 판매 가능 |
| Suspended | `INACTIVE` | 판매 중단 |
| Archived | `ARCHIVED` | 1:1 |
