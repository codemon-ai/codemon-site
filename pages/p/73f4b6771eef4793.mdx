## 2. v2의 장점 (v1 대비)

### ⚡ 속도
- **Vercel 엣지 네트워크** → 전세계 CDN, 서울 리전 우선
- Next.js 서버 컴포넌트 → 초기 로딩 대폭 개선
- v1(EC2 싱글 서버) vs v2(서버리스 자동 스케일링) → 트래픽 급증 시에도 안정

### 🏗️ 안정성
- EC2 수동 관리 제거 → Vercel 자동 배포/롤백
- Supabase 매일 자동 백업
- 서버리스 = 서버 다운 걱정 없음

### 🚀 적용(개발) 속도
- 소규모 개발팀 유지보수 최적화 스택
- NestJS 230개 엔드포인트 → Next.js API Routes로 대폭 간소화
- Drizzle ORM (35KB, 제로 디펜던시) → TypeORM 대비 가볍고 빠름
- AI 코딩 에이전트 활용으로 팀 생산성 극대화

### 🔧 레거시 문제 18개 전면 해결
- Order 잔재 코드 완전 제거
- 상태 관리 이중화 해소
- 무통장입금 별도 테이블 → 결제 통합
- 상품 이중 상태(isActive+status) → status 단일화
- 차량 필드 오염 → 별도 엔티티 분리

### 💡 신규 기능
- **비회원 예약** 정식 지원 (v1은 userId 필수)
- **예약 상태 이력 추적** (누가, 언제, 왜 변경했는지)
- **통합 견적서** (여러 예약 → 1개 인보이스)
- **상품별 취소 수수료 정책** (이용일 기준 차등)
- **부킹 정보 문서** (공급처 전용, v1에는 아예 없었음)

### 🎨 엣지(Edge) 적용 — 현재 30%
- 미들웨어 + 정적 자산에 엣지 적용 중
- 나머지 API는 서울 리전 서버리스 (DB 근접)
- 향후 캐싱 전략으로 더 확대 가능

## 4. 단점 / 솔직한 이야기

### 결제가 아직 단순함
- 토스페이먼츠 연동 미완 (v2 Phase 2)
- 현재: 카드/무통장/가상계좌 3가지만 계획
- v1의 CASH, PAYPAL 제거 (실제 미사용)

### 디자인 갭 차이
- v1의 피그마 대비 갭 분석 결과:
  - 🟢 정상 29% | 🟡 불완전 31% | 🔴 미구현 32% | 🟠 오작동 4%
  - **심각 화면 3개**: 상태 트리거(A-05), 결제완료(A-09), 예약취소(A-12)
- v2는 이 갭을 전부 해결하는 방향으로 설계
- 단, **완전히 피그마대로 구현하려면 클라이언트 확인이 필요한 미결정 사항 28개 존재**

### 가격 갭 차이 극복
- v1에서 이중 통화(원+바트) 표시가 불완전했음
- v2는 환율 관리 + 통화 표시 정상화 설계
- 하지만 환율 프로바이더 1개로 통합 (v1은 3개) → 단순화 vs 안정성 트레이드오프

### v2에서 이미 해결된 것 (v1 이슈와 매핑)

v2 재구축 과정에서 다음 레거시 문제들이 **설계 단계에서 원천 해결**됨:

| v1 문제 | v2 해결 방법 |
|---------|-------------|
| Order 잔재 (orderId 3곳) | 완전 제거, Booking 단일 체계 |
| 상태 이중관리 (enum vs constants) | BookingStatus 단일 enum |
| BankTransfer 독립 테이블 | Payment.method=BANK_TRANSFER 통합 |
| Product isActive+status 이중 | status 단일 필드 |
| Vehicle 7개 nullable 오염 | VehicleProduct 별도 엔티티 |
| 상태 이력 미추적 | BookingStatusHistory 전면 도입 |
| 비회원 예약 불가 | userId nullable + guestToken |
| PDF 스텁 | React Email + react-pdf 설계 |
| 사이트설정 백엔드 없음 | site_settings 테이블 + API |
| 골프 3-way 불일치 | Zod 스키마 + JSONB 단일 소스 |

## 7. 비정형 데이터 대처법 — PDF 가격 분석 Agent 💡

### 문제
- 공급처(업체)에서 제공하는 **복잡한 PDF에 상품 가격**이 적혀있음
- 포맷이 업체마다 다르고, 시즌/객실타입/인원수 등 조건별 가격이 뒤섞여 있음
- 수작업으로 가격을 뽑아서 상품에 입력하는 게 큰 부담

### 제안: PDF 가격 분석 Agent

```
공급처 PDF 업로드
      ↓
  AI Agent가 PDF 파싱
      ↓
  가격 테이블 자동 추출
      ↓
  관리자 확인/수정
      ↓
  상품 가격에 자동 반영
```

#### 구현 아이디어
- **PDF 파싱**: pdf.js 또는 react-pdf로 텍스트 추출
- **AI 분석**: Claude/GPT API로 비정형 텍스트 → 구조화된 가격 데이터
  - 입력: PDF 원문 텍스트
  - 출력: `[상품명, 객실타입, 시즌, 가격_원화, 가격_바트, 조건]`
- **검증 UI**: 추출된 가격을 테이블로 보여주고, 관리자가 확인/수정 후 적용
- **학습**: 같은 공급처의 PDF는 포맷이 비슷하므로 프롬프트 템플릿을 공급처별로 저장

#### 예상 효과
- 가격 입력 시간: **수시간 → 수분**
- 오타/실수 감소
- 시즌 변경 시 빠른 가격 업데이트

#### 기술 스택
- Vercel AI SDK + Claude API (또는 OpenAI)
- Supabase Storage (PDF 보관)
- 공급처별 프롬프트 템플릿 관리

> ⚠️ v2 Phase 3 이후 또는 별도 프로젝트로 진행 권장. MVP에는 포함하지 않되, 로드맵에 넣어두면 좋음.
