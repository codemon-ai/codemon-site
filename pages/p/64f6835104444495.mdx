# Phase 0-3: Order 도입 영향 분석

> **작성일**: 2026-02-25
> **목적**: Order 테이블 도입 시 기존 코드에서 변경이 필요한 지점 식별

---

## 1. 핵심 변경: FK 관계 재설계

### Before (현재)
```
Invoice ──→ (독립, bookingId 간접 연결)
Booking ──→ products, invoiceId(FK)
Payment ──→ bookingId(FK)
Voucher ──→ bookingId(FK)
```

### After (Order 도입)
```
Order ──→ users
  ├── Booking 1 ──→ orderId(FK), products, suppliers
  ├── Booking 2 ──→ orderId(FK), products, suppliers
  └── Booking 3 ──→ orderId(FK), products, suppliers

Invoice ──→ orderId(FK)  ← Order 기준 발송
Payment ──→ orderId(FK)  ← Order 기준 결제
Voucher ──→ bookingId(FK) ← 벤더 확정된 Booking별 발송 (유지)
```

---

## 2. 영향받는 API Routes

### 🔴 전면 수정 필요 (새 로직)

| API | 현재 | 변경 |
|-----|------|------|
| `POST /api/bookings` | Booking 직접 생성 | **Order 생성 → Booking 배열 생성** |
| `POST /api/cart` → checkout | Cart → Booking | **Cart → Order → Bookings** |
| `POST /api/payments/request` | bookingId 기준 | **orderId 기준** |
| `POST /api/payments/confirm` | bookingId 기준 | **orderId 기준** |
| `POST /api/payments/webhook` | bookingId → status | **orderId → Order status** |
| `GET /api/bookings/guest-lookup` | reservationCode | **orderNumber로 조회** |
| `POST /admin/bookings/[id]/invoice` | bookingId 기준 | **orderId 기준, 다중 Booking 묶어서** |
| `POST /admin/bookings/[id]/status` | booking status | **Order status + Booking vendor status** |

## 5. 리스크 및 권장 순서

### 리스크

1. **Invoice FK 변경**: bookingId → orderId 마이그레이션 필요 (기존 데이터)
2. **Payment FK 변경**: bookingId → orderId (또는 둘 다 유지)
3. **단일 상품 주문**: Order에 Booking 1개만 있는 경우 오버헤드
4. **상태 동기화**: Order 상태 ↔ Booking vendorStatus 자동 연동 로직 복잡

### 권장 구현 순서

```
1단계: DB 스키마 (orders, 기존 FK 추가, 신규 테이블)
2단계: Shared Zod 스키마 (Order, 변경된 Booking/Payment)
3단계: Admin API + 페이지 (Order 관리가 먼저)
4단계: Web API + 페이지 (고객 주문 플로우)
5단계: 입금 매칭 + Change Request (부가 기능)
6단계: 패키지 상품 (상품 도메인 확장)
```
