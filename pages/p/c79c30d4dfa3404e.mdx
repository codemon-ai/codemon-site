# BlueMango v1 갭 해결 가능성 분석

> **목적**: v2 갭 분석에서 나온 핵심 갭들을 v1 코드베이스에서 해결할 수 있는지 판단
> **작성일**: 2026-02-23
> **코드베이스**: `bluemango-plan` (branch: `dev`)

---

## 1. v1 아키텍처 요약

### 프로젝트 구조 (모노레포 — 3앱)

```
bluemango-plan/
├── backend/          ← NestJS 10 (admin-api :3001 + service-api :3002)
├── frontend/         ← Next.js (사이트, pnpm)
├── admin-site/       ← Vite + React 18 + Refine + Ant Design (어드민, Vite)
└── package.json      ← 루트 (yarn 4)
```

| 구성 요소 | 스택 |
|-----------|------|
| **백엔드** | NestJS 10 + TypeORM 0.3 + PostgreSQL (AWS RDS) |
| **프론트(사이트)** | Next.js + React Query + TailwindCSS + Toss Payments |
| **어드민** | Vite + React 18 + Refine (@refinedev) + Ant Design 5 |
| **배포** | EC2 올인원 (PM2) + nginx 리버스프록시 |
| **인프라** | S3(이미지), SES(이메일), Sentry(모니터링), Redis(캐시) |

> 💡 **핵심 특징**
> - 백엔드가 **admin-api**와 **service-api** 2개로 분리됨 (같은 entity 공유)
> - 어드민은 **Refine 프레임워크** 기반 (REST data provider)
> - 프론트(사이트)는 Next.js — Vercel 이전 비교적 용이

## 3. Vercel 이전 분석

### 프론트(사이트) → Vercel

| 항목 | 상태 | 비고 |
|------|------|------|
| Next.js 기반 | ✅ 바로 가능 | 이미 next.config.mjs 설정 완비 |
| API 통신 | ⚠️ CORS 설정 필요 | 현재 nginx 프록시 → 직접 API 호출로 변경 필요 |
| 환경변수 | ⚠️ env-cmd 사용 중 | Vercel 환경변수로 이전 필요 |
| 이미지 최적화 | ✅ | Next.js Image 이미 사용, S3 remote patterns 설정됨 |
| standalone 모드 | ⚠️ 주석 처리됨 | `output: 'standalone'` 활성화 필요 |
| Sentry | ✅ | 이미 @sentry/nextjs 설정됨 |

> ✅ **프론트(사이트) Vercel 이전: 1~2일 작업**
> - env-cmd → Vercel 환경변수 이전
> - CORS origin에 Vercel 도메인 추가
> - API base URL 환경변수화 (EC2 백엔드 직접 호출)

### 어드민 → Vercel

| 항목 | 상태 | 비고 |
|------|------|------|
| Vite + React SPA | ✅ 가능 | Vercel은 Vite 지원 |
| 라우팅 | ⚠️ SPA fallback 필요 | `vercel.json`에 rewrite 규칙 추가 |
| API 통신 | ⚠️ CORS | admin-api 엔드포인트에 Vercel 도메인 허용 |
| 인증 | ✅ | JWT 기반, 도메인 무관 |

> ✅ **어드민 Vercel 이전: 0.5~1일 작업**
> - `vercel.json` SPA fallback 설정
> - CORS 도메인 추가
> - 환경변수 이전

### 백엔드 (EC2 유지)

변경 사항:
- CORS 설정에 Vercel 프론트/어드민 도메인 추가
- nginx에서 프론트 서빙 제거 (API만 프록시)
- SSL 인증서는 Vercel이 자동 처리하므로 API 도메인만 관리

## 4. 어드민 교체 vs 수정

### v1 어드민 코드 평가

| 항목 | 평가 |
|------|------|
| **프레임워크** | Refine + Ant Design — 어드민에 적합한 선택 |
| **코드 품질** | 중간 — 페이지 다수 존재 (40+), 기본 CRUD 잘 갖춤 |
| **상태관리** | Refine 내장 (useList, useTable 등) — 일관성 있음 |
| **페이지 수** | 약 25개 카테고리 (booking, product, golf, hotel, car, user, dashboard 등) |
| **문제점** | 일부 페이지 스텁/미완성, console.log 잔재, 타입 불완전 |

### 판단

> ⚠️ **결론: v1 어드민 수정이 합리적**
> 
> **새로 만들지 말 것을 권장하는 이유:**
> 1. 40+ 페이지를 새로 만드는 건 **최소 2~3개월** 소요
> 2. 갭 11개 수정은 **20~30일**이면 충분
> 3. Refine + Ant Design은 어드민에 충분히 좋은 스택
> 4. 백엔드 API가 동일하므로 어드민만 교체해도 이득이 거의 없음
> 
> **만약 새로 만든다면:**
> - Next.js + shadcn/ui 등으로 만들면 UX는 좋아지나
> - 기존 40+ 페이지 재구현 필요 → ROI 매우 낮음
> - 백엔드도 함께 리팩토링하지 않으면 의미 반감

## 5. 최종 권장안

> 📋 **권장: v1 수정 + Vercel 이전 (하이브리드)**

### 전략

```
Phase 1 (1~2주): Vercel 이전
├── 프론트(사이트) → Vercel 배포
├── 어드민 → Vercel 배포
├── 백엔드 → EC2 유지 (CORS만 수정)
└── nginx → API 프록시만 남기기

Phase 2 (3~4주): v1 갭 수정
├── 🔴 미구현 8개 해결
├── 🟠 오작동 3개 수정
└── QA + 안정화

Phase 3 (이후): 점진적 개선
├── 백엔드 리팩토링 (필요시)
├── v2 설계 부채 해결 (JSONB 등)
└── Phase 2/3 기능 (정산, 알림 등)
```

### v2 진행 vs v1 수정 비교

| 항목 | v1 수정 | v2 신규 |
|------|---------|---------|
| **갭 해결 기간** | 4~6주 | 3~4개월+ |
| **리스크** | 낮음 (기존 코드 수정) | 높음 (DB 마이그레이션, Supabase URL 미확보) |
| **배포 환경** | Vercel 이전 가능 | Vercel 네이티브 |
| **기술 부채** | 일부 잔존 | 깔끔한 시작 |
| **즉시 운영** | ✅ 가능 | ❌ MVP 완성까지 대기 |

> 🚨 **v2 블로커 상존**
> Supabase DATABASE\_URL 미수신 상태 — v2 개발 시작 자체가 불가.
> v1 수정으로 **즉시 운영 가능한 상태**를 만드는 것이 현실적.

### 최종 의견

1. **v1 수정 + Vercel 이전**으로 당장의 갭 해결 (4~6주)
2. v2는 Supabase 접근 확보 후 **장기 계획**으로 진행
3. 어드민은 **새로 만들지 않고** v1 수정으로 충분
4. 프론트(사이트)와 어드민 모두 Vercel 이전은 **2~3일**이면 완료
