**왜 Anthropic이 성공가도를 달리는가? — B2B 플레이북 분석**

수많은 AI 플레이어 중 Anthropic이 14개월 만에 ARR $1B → $14B (10배)를 달성한 이유는 **기술이 아니라 전략**입니다. 스타트업 대표가 배워야 할 7가지:

| 전략 | 핵심 | 스타트업 교훈 |
|------|------|------------|
| **API-First** | ChatGPT가 웹 인터페이스로 갔을 때, Claude는 API를 먼저 만듦 | 기업 시스템에 녹아드는 제품이 이긴다 |
| **적은 사용자, 높은 매출** | 사용자 1,900만(OpenAI의 1/40), 하지만 사용자당 월 $211(OpenAI $25) | 개인이 아닌 **조직의 문제**를 풀면 단가가 다르다 |
| **Usage-Based 진입** | 개발자가 신용카드로 시작 → 팀 확산 → 기업 계약 | 영업 없이 제품으로 파는 PLG(Product-Led Growth) |
| **코딩 = 킬러 유즈케이스** | Claude Code만 $2.5B ARR, AI 코딩 시장 점유율 54% | 토큰 많이 쓰는 유즈케이스 = 매출 폭발 |
| **100만 명 비공식 영업팀** | AWS(Bedrock), Google(Vertex), Microsoft(Copilot) + 딜로이트 47만 명, 액센추어 3만 명 | 파트너의 영업팀을 내 영업팀으로 만들기 |
| **MCP = 업계 표준 선점** | OpenAI, Google, MS 모두 MCP 채택 → 경쟁사가 따라올 수밖에 없는 구조 | 표준을 만드는 자가 시장을 지배한다 |
| **안전 = 셀링 포인트** | "AI Safety" 미션이 엔터프라이즈 보안 검토를 통과하는 무기 | 미션과 비즈니스를 일치시켜라 |

> **핵심 역설**: OpenAI 최대 투자자인 Microsoft조차 Claude를 Copilot에 기본 탑재. **투자 관계보다 제품 성능이 이긴다**는 시장 신호.

> **총이익률**: Anthropic -94% → 50% (1년 만에 전환) — B2B 단위 경제학이 작동하기 시작한 증거. 2028년 전망 $70B ARR.

**스타트업 대표가 가져갈 것**: "멋진 기술을 만드는 것"이 아니라 "조직의 문제를 풀고, 쓰는 만큼 과금하고, 파트너로 확산시키는 것." 이것이 AI 시대의 B2B 플레이북입니다.

**반대편의 이야기 — "SaaS is Dead"**

AI 도구 시장이 폭발하는 동안, 기존 SaaS 기업들은 무너지고 있습니다.

2026년 2월, Claude Cowork 출시 48시간 만에 소프트웨어 주식 **$2,850억(약 413조원) 증발**. 이후 수 주간 총 **$2조 규모**의 SaaS 주식 가치가 사라졌습니다. 월가에서는 이를 **"SaaSpocalypse(SaaS 대멸종)"**라 부릅니다.

| 기업 | 하락폭 | 이유 |
|------|--------|------|
| **Salesforce** | -28% | AI가 영업 담당자 10명 일을 1명+에이전트로 대체 → 시트 매출 90% 감소 위험 |
| **ServiceNow** | -40% | 가장 큰 낙폭, IT 서비스 자동화 영역을 AI가 직접 수행 |
| **SW ETF (IGV)** | -22% YTD | 2022년 금리 인상 이후 최대 소프트웨어 하락 |

**왜 SaaS가 죽고 있는가?**

| 원인 | 설명 |
|------|------|
| **AI가 UI를 대체** | 메뉴 클릭 대신 AI에게 자연어로 지시 → SaaS의 화면이 필요 없어짐 |
| **시트 모델 붕괴** | 10명이 하던 일을 AI 에이전트가 하면 → 10개 구독이 1개로 줄어듦 |
| **내부 도구 직접 제작** | 바이브 코딩으로 누구나 내부 도구를 만들 수 있어 SaaS 구독 해지 |
| **Switching Cost 하락** | AI가 복잡한 마이그레이션을 쉽게 만들어 SAP, Oracle 떠나기 쉬워짐 |

> 마이크로소프트 CEO 나델라: **"SaaS is dead"** (2024.12)
> IDC 전망: 2028년까지 **70%의 SW 벤더**가 시트 기반 가격을 폐기

**스타트업 대표에게 의미하는 것**: 기존 SaaS에 의존하는 비즈니스 모델은 위험합니다. 반대로, AI 에이전트 기반 서비스를 만들면 이 파도를 탈 수 있습니다. **SaaS 1.0(UI 중심 구독)은 죽고, SaaS 2.0(AI 에이전트 기반 성과형)이 떠오르고 있습니다.**

**실제 예시 — 저도 SaaS를 해지했습니다**

저는 프로젝트 관리 도구(Jira, Notion, Monday)를 쓰다가 직접 만들었습니다. AI와 함께 만든 **codemon-make**라는 내부 도구입니다.

| 비교 | 기존 SaaS (Jira/Monday) | 직접 만든 도구 (codemon-make) |
|------|----------------------|--------------------------|
| 월 비용 | $10~30/인/월 | 서버비 $0 (Vercel 무료) |
| 맞춤화 | 제한적 | 우리 팀 워크플로우에 100% 맞춤 |
| AI 연동 | 별도 플러그인 필요 | 에이전트가 직접 이슈 생성·처리 |
| 제작 기간 | — | AI와 함께 **3일** |

"코드 못 쓰는데 어떻게요?" — 바이브 코딩으로 만들었습니다. 이게 바로 다음 모듈에서 이야기할 내용입니다.

#### M3. AI 팀원의 작동 원리 — 연결, 역할, 매뉴얼

새 직원을 뽑으면 뭘 해줍니까?

| 새 직원에게 하는 것 | AI에게 하는 것 | 개념 |
|------------------|-------------|------|
| ① 회사 시스템 계정을 준다 | 회사 도구에 연결한다 | **MCP** |
| ② 역할을 정해준다 | 담당 업무를 배정한다 | **에이전트** |
| ③ 업무 매뉴얼을 준다 | 반복 업무를 정의한다 | **스킬** |

> "AI도 사람과 똑같습니다. 계정, 역할, 매뉴얼. 이 세 가지가 있으면 일합니다."

이 세 가지를 하나씩 자세히 보겠습니다.

**① MCP = "AI의 USB-C 포트"** — 연결

AI가 회사 시스템에 로그인해서 직접 일하는 표준 프로토콜

- **12,000+** 커뮤니티 서버 (2026.03 기준)
- OpenAI, Google, Microsoft, xAI 모두 채택 → 사실상 유일 표준
- OpenAI가 Assistants API를 2026년 중반 종료 예정 → MCP로 통합

| 연결 | AI가 하는 일 | 스타트업 활용 |
|------|-------------|-------------|
| MCP + **Slack** | 메시지를 읽고, 요약하고, 답장 | 고객 문의 자동 응대 |
| MCP + **Google Calendar** | 일정 확인하고 회의를 잡음 | 스케줄링 자동화 |
| MCP + **GitHub** | 코드를 읽고, PR을 만들고, 리뷰 | 코드 리뷰 자동화 |
| MCP + **Linear/Jira** | 이슈 생성, 상태 추적 | 프로젝트 관리 자동화 |

> **데모 2 — 회사 도구 연결**: MCP로 슬랙을 AI에 연결하고 채널 요약 요청

**② 에이전트 = 역할이 있는 AI 팀원** — 역할

여러 에이전트가 팀을 이루면 부서가 됩니다.

```
대표 (지시)
├── AI 리서처: "경쟁사 분석해줘"
├── AI 에디터: "블로그 글 써줘"
└── AI 개발자: "랜딩페이지 만들어줘"
```

**서브에이전트**: 메인 에이전트가 작업을 쪼개서 전문 에이전트에게 위임. 각각 독립된 컨텍스트에서 작업 후 결과를 보고.

**Agent Teams**: 서브에이전트의 진화형. AI끼리 **서로 직접 소통** 가능 — 팀장이 매번 중계하지 않아도 됩니다.

| **Cursor** | Anysphere | IDE 통합, AI 코드 에디터 1위 |

**③ 스킬 = AI 업무 매뉴얼** — 매뉴얼

반복 업무를 스킬로 정의하면 AI가 자동으로 인식하고 일관된 결과를 냅니다.

사람으로 치면:
- "회의록은 이 양식으로 써" → 다음부터 알아서 함
- "메일 요약은 이렇게 해" → 매번 안 알려줘도 됨

스킬의 3가지 특징:
1. **자동 인식** — "회의록 정리해줘" 하면 스킬이 알아서 발동
2. **일관된 결과** — 매번 양식 설명 안 해도 됨
3. **비개발자도 생성 가능** — Claude Desktop에서 자연어로 3분이면 작성

---

**이 세 가지를 합치면? → 플러그인**

Claude Code에서는 MCP + 스킬 + 훅 + CLAUDE.md를 하나로 묶은 것을 **플러그인(Plugin)**이라 부릅니다. 플러그인 하나를 설치하면 AI가 특정 도메인의 전문가가 됩니다.

```
플러그인 = CLAUDE.md(온보딩) + Skills(매뉴얼) + Hooks(가드레일) + MCP(도구 연결)
```

| 예시 플러그인 | 포함 내용 | 효과 |
|-------------|---------|------|
| Vercel 플러그인 | 배포 스킬 + 빌드 훅 + Vercel MCP | AI가 배포 전문가가 됨 |
| 코드 리뷰 플러그인 | 리뷰 스킬 + 보안 점검 훅 | AI가 시니어 리뷰어가 됨 |
| 블로그 발행 플러그인 | 발행 스킬 + SEO 점검 + CMS 연결 | AI가 퍼블리셔가 됨 |

> "하나의 AI에 플러그인을 꽂으면 전문가가 됩니다. 여러 전문가를 모으면 팀이 됩니다. 그 팀이 바로 다음에 보여드릴 것입니다."

**CLAUDE.md = AI 온보딩 문서**: 프로젝트 규칙, 금지 사항, 선호 방식을 한 번 적어두면 AI가 매번 따름. 플러그인의 핵심 뼈대.

### 3장. "AI만 있으면 다 되는거 아닌가요?" (40분)

#### M5. 시스템이 필요하다 — 하네스 엔지니어링 (20분)

**"AI만 있으면 다 됩니까?" — 안 됩니다.**

"ChatGPT 잘 쓰면 되는 거 아닌가요?" 이 질문을 가장 많이 받습니다. 답은 **"AI는 도구이고, 도구를 잘 쓰려면 시스템이 필요합니다."**

OpenAI Codex 팀이 100만 줄 코드 애플리케이션을 만들었는데, **사람이 코드를 한 줄도 안 썼습니다.** 대신 사람이 한 일은? **AI가 코드를 잘 쓰도록 하는 시스템을 설계**한 겁니다. 이것이 하네스 엔지니어링입니다.

**엔지니어링의 3단계 진화**

| 단계 | 이름 | 비유 | 핵심 |
|------|------|------|------|
| 1단계 | 프롬프트 엔지니어링 | **잘 물어보기** | "이렇게 요약해줘" |
| 2단계 | 컨텍스트 엔지니어링 | **잘 알려주기** | "우리 회사는 이런 곳이야" (CLAUDE.md) |
| 3단계 | **하네스 엔지니어링** | **잘 부리기** | 시스템으로 AI를 제어 (스킬 + 훅 + 검증) |

> "프롬프트를 잘 쓰는 시대는 끝났습니다. 지금은 AI를 **시스템으로 제어**하는 시대입니다."

**하네스의 구성 요소 (Claude Code 예시)**

```
하네스 = CLAUDE.md + Skills + Hooks + 이중검증
         (온보딩)   (매뉴얼)  (가드레일)  (품질관리)
```

- **CLAUDE.md**: "우리 회사는 이런 곳이야" — AI가 프로젝트에 들어올 때 가장 먼저 읽는 규칙서
- **Skills**: "이 일은 이렇게 해" — 반복 업무를 정의해두면 AI가 자동으로 따름
- **Hooks**: "이건 하지 마 / 하고 나서 검증해" — 자동 가드레일

| 훅 | 시점 | 스타트업 활용 예시 |
|----|------|-----------------|
| **PreToolUse** | 도구 실행 전 | 프로덕션 DB 수정 차단, 민감 파일 보호 |
| **PostToolUse** | 도구 실행 후 | 코드 수정 시 자동 포맷팅 + 테스트 |
| **SessionStart** | 세션 시작 | 프로젝트 컨텍스트 자동 로드 |
| **Stop** | 작업 완료 | 슬랙/텔레그램으로 결과 알림 |

> "사람에게 규칙을 반복해서 말해야 합니다. AI에게는 한 번만 설정하면 100% 지킵니다."

**이중 검증 구조**: AI 생성 → AI 검증 → 사람 승인

```
[AI 생성]              [AI 검증]               [사람 승인]
로디가 글 작성  →  누비가 팩트체크 + 검수  →  대표가 최종 확인
                      ↓
                자동 테스트 + 보안 점검
```

**에이전틱 AI의 유행 — Clawdbot → Moltbot → OpenClaw 이야기**

2025년 11월, 오스트리아 개발자가 **Clawdbot**이라는 오픈소스 AI 에이전트를 공개합니다. Anthropic의 상표 이슈로 **Moltbot**, 다시 **OpenClaw**로 이름이 바뀌면서 오히려 더 유명해졌고, **GitHub 역사상 가장 빠르게 성장한 오픈소스 프로젝트** (60일 만에 25만+ 스타)가 됩니다.

왜 폭발적으로 유행했을까?

| 이유 | 설명 |
|------|------|
| **조합의 힘** | MCP로 Slack, 캘린더, 이메일, 브라우저를 연결 → 하나의 AI가 모든 앱을 넘나들며 작업 |
| **누구나 설치** | 오픈소스, 로컬 실행 — 기업 보안 걱정 없이 시작 |
| **실제 업무 수행** | 웹 검색, PDF 요약, 일정 잡기, 이메일 발송까지 자율 실행 |
| **커뮤니티 효과** | Moltbook(에이전트끼리 소통하는 포럼)이 바이럴 촉발 |

> **교훈**: AI 하나가 대단한 게 아니라, **MCP + 스킬 + 훅으로 조합하는 시스템**이 대단한 겁니다. OpenClaw가 증명한 것은 "연결의 힘"입니다.

단, 보안 이슈도 있었습니다 — 21,000+ 인스턴스가 인증 없이 노출. **하네스 없이 에이전트를 풀어놓으면 위험**합니다.

**각 AI 플랫폼의 데이터 보안 정책**

| 플랫폼 | 데이터 학습 | 비고 |
|--------|-----------|------|
| Claude (API/Pro) | ❌ 미학습 | 엔터프라이즈 추가 보장 |
| GPT (API) | ❌ 미학습 | ChatGPT 무료는 학습됨 |
| Gemini (API) | ❌ 미학습 | 무료 버전은 학습됨 |
| Grok (API) | ❌ 미학습 | X 데이터는 별도 |

> "API를 쓰면 어떤 AI든 데이터를 학습하지 않습니다. **하네스로 보안까지 관리**하세요."

> **데모 5 — 바이브 코딩**: 코드 한 줄 없이 랜딩페이지 만들기
>
> **데모 6 — 자동 품질 점검**: AI가 사이트를 테스트하고 버그 리포트를 작성하는 과정

**실제 예시 — Paperclip: AI 에이전트 회사의 컨트롤 플레인**

"지금 직접 만들고 있는 것을 보여드리겠습니다."

**Paperclip**은 AI 에이전트들로 구성된 **자율 회사(Autonomous Company)**를 운영하기 위한 컨트롤 플레인입니다.

```
대표 (사람, 1명)
├── CEO 에이전트 — 전략 수립, 리뷰, 우선순위 조정
│   ├── CTO — 개발팀 관리, 코드 리뷰
│   │   ├── 프론트엔드 엔지니어
│   │   └── 백엔드 엔지니어
│   ├── CMO — 마케팅, 콘텐츠, SNS
│   │   └── 콘텐츠 에디터
│   └── CFO — 비용 추적, 예산 관리
```

**핵심 구조**

| 개념 | 설명 | 왜 필요한가 |
|------|------|-----------|
| **회사 = 단위** | 하나의 Paperclip에서 여러 회사를 운영 | 외주 프로젝트별로 AI 팀을 분리 |
| **에이전트 = 직원** | 각 에이전트에 역할, 상사, 부하, 예산 배정 | 조직도가 있어야 일이 흐른다 |
| **태스크 = 목표 연쇄** | 모든 작업이 회사 목표까지 연결 | "왜 이 일을 하는가?" 항상 추적 가능 |
| **어댑터 = 실행** | Claude Code, Codex, OpenClaw 등 무관 | AI 도구에 종속되지 않음 |

> "이것이 '5명이 50명을 이기는' 최종 형태입니다. 사람 1명이 AI 회사의 이사회 역할을 하고, 나머지는 AI가 조직도 안에서 스스로 일합니다."
