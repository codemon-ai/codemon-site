# Module 2: Claude 생태계 한눈에 (15분)

> 발표 목표: Claude의 3단계 진화를 비개발자도 이해할 수 있게 설명하고, "AI 비서 → AI 팀"이라는 개념 전달

---

## AI의 3단계 진화

```
1단계: Claude (대화)         → AI 비서와 대화하기
2단계: Claude Code (코딩)    → AI 비서가 일을 실행한다
3단계: Claude Cowork (업무)  → AI 팀이 협업한다
```

### 비개발자용 설명

| 단계 | 비유 | 할 수 있는 것 |
|------|------|-------------|
| Claude | 똑똒한 인턴 | 질문에 답변, 문서 요약, 아이디어 브레인스토밍 |
| Claude Code | 실행력 있는 팀원 | 코드 작성, 파일 수정, 테스트 실행, 배포 |
| Claude Cowork | AI 팀 | 여러 AI가 역할 분담해서 프로젝트를 진행 |

### 개발자용 설명

| 단계 | 기술 | 도구/기능 |
|------|------|----------|
| Claude API | LLM 호출 | Messages API, Tool Use |
| Claude Code | 코딩 에이전트 | 터미널 네이티브, 1M 토큰 컨텍스트, Extended Thinking |
| Claude Cowork | 멀티 에이전트 | Subagent, Agent Teams, Agent SDK |

## MCP — AI가 도구를 쓰는 표준

**MCP (Model Context Protocol)**: AI 에이전트가 외부 도구/서비스에 접근하는 표준 프로토콜

> Anthropic이 "AI 앱의 USB-C 포트"라고 설명

### 비개발자 비유

"지금까지 AI는 '대화만 하는 직원'이었다. MCP를 연결하면 '회사 시스템에 로그인해서 직접 일하는 직원'이 된다."

예시:
- MCP + Slack → AI가 슬랙 메시지를 읽고 답장
- MCP + Google Calendar → AI가 일정을 확인하고 회의를 잡음
- MCP + GitHub → AI가 코드를 읽고, PR을 만들고, 리뷰

### 현황

- **1,000개+** 커뮤니티 서버
- OpenAI, Google, Microsoft 모두 채택 → 사실상 유일 표준
- OpenAI가 Assistants API를 2026년 중반 종료 예정 → MCP로 통합

## 서브에이전트 — 일을 나눠주는 AI

Claude Code 안에서 특정 작업을 전담하는 **격리된 AI 인스턴스**.

```
메인 에이전트 (지시)
├── 서브에이전트 A: "이 코드 리뷰해줘"
├── 서브에이전트 B: "관련 문서 찾아줘"
└── 서브에이전트 C: "테스트 작성해줘"
```

각 서브에이전트는:
- 자기만의 컨텍스트 윈도우 (다른 에이전트와 격리)
- 전문 역할에 맞는 도구 권한
- 결과를 메인 에이전트에게 보고

기본 제공 타입:

| 타입 | 역할 | 에이전시 활용 예시 |
|------|------|-----------------|
| `general-purpose` | 범용 작업 | 클라이언트 요구사항 정리 |
| `Explore` | 코드베이스 탐색 | 기존 프로젝트 파악 |
| `Plan` | 아키텍처 설계 | 프로젝트 설계안 작성 |
| `code-reviewer` | 코드 리뷰 | QA 자동화 |

## Agent Teams — AI끼리 팀을 짠다

2026년 2월 출시. 서브에이전트의 진화형.

**핵심 차이**: 서브에이전트는 메인을 거쳐야 소통하지만, Agent Teams의 Teammate는 **서로 직접 소통** 가능

```
[서브에이전트]                  [Agent Teams]
메인 ←→ A                    Lead ←→ A ←→ B
메인 ←→ B                              ↕
(A와 B는 직접 소통 불가)             C
                            (모두 직접 소통 가능)
```

실제 사용 예시:

```
> "로그인 페이지를 리디자인해줘. 프론트, API, 테스트 동시에."

Team Lead가:
→ Frontend Agent: UI 컴포넌트 수정
→ Backend Agent: API 엔드포인트 업데이트
→ Test Agent: E2E + 단위 테스트 작성
각 에이전트가 독립적으로 작업 후 결과 통합
```

## 스킬(Skills) — AI에게 업무 매뉴얼을 가르치다

### 스킬이란?

**한 마디로: AI에게 가르쳐둔 업무 매뉴얼.**

사람으로 치면:
- 신입한테 "회의록은 이 양식으로 써" → 다음부터 알아서 함
- 인턴한테 "메일 요약은 이렇게 해" → 매번 안 알려줘도 됨

AI 스킬도 같다. SKILL.md 파일 하나 만들어두면:
1. **자동으로 인식** — "회의록 정리해줘" 하면 스킬이 알아서 발동
2. **일관된 결과** — 매번 양식 설명 안 해도 됨
3. **재사용 가능** — 모든 프로젝트에서 같은 스킬을 사용

### 두 가지 맛: Cowork 스킬 vs Claude Code 스킬

| | Cowork 스킬 | Claude Code 스킬 (SKILL.md) |
|---|---|---|
| **대상** | 비개발자 (대표/PM/기획자) | 개발자 |
| **만드는 법** | Claude Desktop에서 자연어로 작성 | `.claude/skills/` 폴더에 마크다운 파일 |
| **실행** | Claude Desktop 앱 | 터미널에서 `/스킬이름` |
| **도구** | 제한적 (대화 범위 내) | 파일 시스템, Git, API, 터미널 전체 |
| **자동화** | 스케줄 태스크 | cron, 파이프라인, 훅 연동 |
| **코딩 필요** | 없음. 한국어로 3줄이면 됨 | 마크다운 + YAML frontmatter |

**비개발자에게**: "Claude Desktop에서 3분이면 만들 수 있다. 코딩 필요 없음."

### Skills의 진화: Commands → Skills 2.0

Claude Code 초기에는 **커스텀 명령어(Commands)** 시스템이었다:

```
[Before — Commands 1.0]
.claude/commands/review.md     ← 단순 마크다운 파일
                               ← /review 로만 수동 호출
                               ← 인자 처리 없음
                               ← AI가 자동 판단 불가
```

현재는 **스킬(Skills 2.0)** 로 통합 진화:

```
[After — Skills 2.0]
.claude/skills/review/
├── SKILL.md                   ← YAML 메타데이터 + 워크플로우 정의
├── references/                ← 참고 자료 번들
├── examples/                  ← 예시 파일
└── scripts/                   ← 보조 스크립트
```

| 기능 | Commands 1.0 | Skills 2.0 |
|------|-------------|-----------|
| 호출 방식 | `/명령어`로 수동만 | 수동 `/스킬` + **AI가 맥락 보고 자동 발동** |
| 인자 전달 | 없음 | `$ARGUMENTS` 플레이스홀더 |
| 동적 컨텍스트 | 없음 | 셸 커맨드로 실시간 데이터 주입 |
| 도구 제한 | 없음 (전체 접근) | `allowed-tools`로 최소 권한 적용 |
| 실행 환경 | 메인 세션 | `context: fork`로 격리 실행 가능 |
| 서브에이전트 위임 | 불가 | `agent: Explore` 등으로 서브에이전트 실행 |
| 부작용 제어 | 없음 | `disable-model-invocation: true` |
| 파일 참조 | 없음 | `@파일경로`로 자동 로드 |
| 표준 | 독자 규격 | **Agent Skills 오픈 표준** (agentskills.io) |
| 기존 호환 | — | `.claude/commands/` 파일도 그대로 동작 |

### SKILL.md 실전 예시

```yaml
---
name: fix-issue
description: "GitHub 이슈를 분석하고 수정한다.
  '/fix-issue 1234' 형태로 호출."
disable-model-invocation: true
allowed-tools: ["Read", "Edit", "Bash(gh *)"]
---

GitHub 이슈 $ARGUMENTS를 수정한다.

1. `gh issue view`로 이슈 상세 확인
2. 코드베이스에서 관련 파일 검색
3. 수정 구현
4. 테스트 작성 및 실행
5. 커밋 + PR 생성
```

```yaml
---
name: pr-summary
description: "PR 변경 사항을 요약한다"
context: fork
agent: Explore
---

## PR 컨텍스트
- PR diff: (gh pr diff 실행)
- 변경 파일: (gh pr diff --name-only 실행)

이 PR의 변경 사항을 요약해줘.
```

### 핵심 포인트: 왜 이게 에이전시에 중요한가?

1. **반복 업무의 표준화**: 프로젝트마다 같은 QA 체크, 코드 리뷰, 리포트 → 스킬로 한 번 만들면 끝
2. **팀원 온보딩 비용 절감**: 새 개발자가 `/fix-issue`, `/code-review` 스킬만 알면 바로 생산성 발휘
3. **품질 일관성**: 사람마다 다른 리뷰 품질 → 스킬이 기준선을 보장
4. **오픈 표준**: Agent Skills 표준을 따르므로 다른 AI 도구와도 호환 가능

## Agent SDK — 프로그래밍적 제어 (개발자 대상)

Agent SDK는 Claude Code의 기능을 Python/TypeScript로 직접 제어.

**에이전시에 의미하는 것**:
- 반복되는 프로젝트 셋업을 코드로 자동화
- 에이전트 파이프라인을 프로그래밍적으로 구성
- 클라이언트별 맞춤 워크플로우 구축

```python
# 예: QA 전용 에이전트 (Agent SDK)
agent = ClaudeAgent(
    model="claude-sonnet-4-6",
    tools=[code_search, test_runner],
    system_prompt="변경된 코드의 테스트를 작성하고 실행해."
)
```
