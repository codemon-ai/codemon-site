LLM 위키 = 나만의 지식노트

STEP 1지식노트 만들고 클로드에 연결60분

목표 · 마크다운 볼트를 만들고, 클로드가 그 폴더를 직접 읽고 정리하게 한다.
왜 'LLM 위키'인가 — RAG와 뭐가 다른가

흔한 방식(RAG·파일 업로드)은 질문할 때마다 원본에서 관련 조각을 찾아 답을 만듭니다. LLM 위키는 클로드가 소스를 읽고 마크다운 페이지로 남긴 뒤, 새 소스가 들어올 때마다 다시 갱신하는 방식입니다. 요약·링크·모순 표시가 파일로 남아서 다음 질문에 재사용됩니다.

수강생은 소스를 넣고 질문하고, 클로드는 요약·연결·정리 작업을 맡습니다. (안드레아 카파시의 LLM Wiki 패턴 기반.)

A. Obsidian 볼트 & 첫 노트

  1. Obsidian 실행 → "Create new vault" 옆 Create
  2. Vault name 입력 → Browse로 저장 위치 지정 → Create
  3. 첫 노트: Cmd/Ctrl + N → 파일명 → Enter → 본문 작성
    마크다운 지원. 텍스트 선택 후 Cmd/Ctrl + B로 볼드.
  4. 백링크 연결: 본문에 [[ 입력 → 노트 선택
    별칭 [[노트|보일이름]], 헤딩 링크 [[노트#헤딩]].
볼트는 그냥 .md 파일 폴더입니다 — 별도 DB·변환 없이 클로드가 원본 그대로 읽습니다(그래서 'AI가 읽기 좋은' 노트).

좋은 노트의 기준

나쁜 시작좋은 시작
제목 없는 긴 메모, 출처 없음, 연결 없음제목 + 출처 + 핵심 3줄 + 관련 노트 [[링크]]
채팅 답변을 복사해두고 끝답변을 새 페이지로 저장하고 index.md, log.md 갱신

B. 클로드 데스크탑에 볼트 폴더 연결 (파일시스템 MCP)

기본은 방법 1(Extensions 원클릭)입니다. JSON 방식은 원클릭이 안 되거나 수동 서버가 필요할 때만 씁니다.

  1. 방법 1 — 원클릭: Settings > Extensions → "Browse extensions"
    Anthropic 검수 도구 디렉토리에서 파일시스템 도구 설치(브라우저 확장처럼). JSON 편집 불필요.
  2. 방법 2 — JSON(고급/대체): Claude 설정의 Developer/고급 설정에서 config 편집
    UI 명칭은 버전에 따라 달라질 수 있습니다. claude_desktop_config.json에 아래 내용으로 볼트 경로를 지정합니다. 이 방식은 npx를 쓰므로 Node.js가 필요합니다.
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem",
               "/내/Obsidian/볼트/경로"]
    }
  }
}
  1. Claude Desktop 완전 종료 후 재시작
  2. 입력창 우하단 MCP 인디케이터 확인
    안 뜨면: 완전 재시작 / 권한 / 절대경로 확인. JSON 방식이면 문법과 Node.js 설치도 확인.
수업 기본 경로는 공식 Desktop Extensions입니다. 수동 JSON 경로는 UI가 바뀔 수 있으니, 당일 Claude Desktop 설정 화면을 기준으로 확인합니다.

C. 클로드에게 시키기 — 예시 프롬프트

공식 문서 검증 예시
"What work-related files are in my downloads folder?"
"Please organize all images on my desktop into a new folder."
응용 예시 (작성자 제안 — 볼트 특화)
"내 볼트 전체를 읽고 주제별 인덱스 노트를 만들어줘."
"이번 주에 쓴 노트 5개를 요약해줘."
"관련 있는 노트끼리 [[백링크]]로 연결해줘."
클로드가 볼트의 파일을 읽고(승인 후) 새 노트를 만들거나 요약하면 1차 성공.

D. 위키를 계속 쓰게 만드는 3가지 — 스키마·인덱스·운영

노트를 모으는 데서 그치지 않고, 새 소스가 들어왔을 때 어디에 넣고 어떻게 갱신할지 정해둡니다. 아래 셋만 갖추면 입문용 LLM 위키로 충분합니다.

① 스키마 — 볼트 루트에 CLAUDE.md 한 장

위키 구조·규칙·작업 순서를 적은 설정 파일. 매 세션 클로드가 먼저 읽고 같은 규칙으로 작업하게 만듭니다.

# 내 지식 위키 — 유지 규칙 (CLAUDE.md)
- raw/   : 원본 소스(클로드는 읽기만, 절대 수정 금지)
- pages/ : 클로드가 만드는 위키 페이지(요약·인물·개념)
- index.md : 전체 페이지 목록 + 한 줄 요약
- log.md   : 한 일 기록(시간순)

# 새 소스가 들어오면(Ingest):
1) raw/ 소스를 읽고 핵심을 pages/ 에 요약 페이지로 저장
2) 관련 페이지를 [[백링크]]로 연결, 모순되면 표시
3) index.md 갱신 + log.md 에 한 줄 추가
② 인덱스·로그 — index.md + log.md
  1. index.md = 위키 목차
    모든 페이지를 [링크 + 한 줄 요약]으로. 클로드가 답할 때 먼저 여기를 읽고 관련 페이지로 들어갑니다 — 작은 규모에선 이것만으로 검색이 충분합니다.
  2. log.md = 시간순 기록
    한 줄씩 누적: ## [2026-06-27] ingest | 글 제목. 언제 뭘 넣었는지 추적되고 grep으로도 찾힙니다.
③ 3대 작업 — Ingest · Query · Lint (예시 프롬프트)
# Ingest(소스 투입) — 새 글/메모 1개 정리
"raw/ 에 넣은 이 글을 읽고 요약 페이지를 만들고,
 관련 노트에 [[백링크]] 연결한 뒤 index.md·log.md를 갱신해줘."

# Query(질문) — 답을 위키에 되돌려 저장(=복리)
"내 위키로 'X와 Y의 차이'를 표로 정리하고,
 그 표를 새 페이지로 저장해줘."

# Lint(건강검진) — 주기적으로
"위키에서 서로 모순되는 내용·낡은 주장·
 아무도 안 잇는 고아 페이지를 찾아 고칠 것을 제안해줘."
포인트: 질문(Query) 결과도 위키에 다시 저장하면 탐색이 채팅 기록으로 사라지지 않고 다음 작업에 재사용됩니다.
CLAUDE.md를 둔 볼트에서 클로드가 새 소스를 인제스트하고 index.md·log.md까지 갱신하면 성공. 확인할 것: 계속 갱신되는 내 LLM 위키(볼트 + CLAUDE.md + 정리 노트)
Skill · MCP · Cowork

STEP 2도구를 붙이고 일을 맡긴다60분

목표 · Skill·MCP·Cowork 개념을 손으로 체감하고, 폴더 작업을 위임해 실파일을 만든다.

A. 세 단어 정리 — Skill / MCP / Cowork

순서는 간단합니다. Skill은 일하는 방법, MCP·Connector는 도구에 닿는 통로, Cowork는 그 방법과 통로로 실제 일을 맡기는 화면입니다.

개념비유오늘 하는 일
Skill업무 매뉴얼PDF·엑셀·PPT·Word 결과물 만들기
MCP / Connector도구 연결선로컬 폴더와 외부 도구에 접근
Cowork일 맡기는 작업실폴더 단위 작업을 에이전트에게 위임

B. 내장 Skill 체감 (PDF·엑셀·PPT·Word)

claude.ai에는 pptx·xlsx·docx·pdf 작업용 Skill이 내장돼 별도 설정 없이 자동 동작합니다. 그냥 자연어로 요청:

"이 데이터로 엑셀 매출표랑 차트 만들어줘."
"이 내용으로 발표용 PPT 10장 만들어줘."
"이 PDF 핵심만 요약해서 Word 보고서로."
커스텀 Skill은 zip으로 올릴 수 있습니다. 공식 문서 기준 경로는 claude.ai의 Settings > Features 계열이지만, UI 명칭은 바뀔 수 있으므로 수업일 화면에서 다시 확인합니다.

C. 공식/공식 마켓플레이스 도구 — 무엇을 더 붙일 수 있나

아래는 실제 검색으로 공식 문서 또는 Claude Code 공식 마켓플레이스 등록을 확인한 도구입니다. 4시간 수업의 필수 경로는 아니고, 준비된 수강생이나 강사 시연에서 사용합니다.

도구언제 쓰나수업에서의 위치
Superpowers브레인스토밍, 서브에이전트 개발, 코드 리뷰, 체계적 디버깅, red/green TDD 흐름Claude Code 심화 Skill. 서드파티지만 공식 마켓플레이스에 등록
Playwright CLI + Skills에이전트가 명령어로 브라우저를 열고 클릭·입력·스크린샷·간단 점검 수행배포 URL 확인용. 코딩 에이전트에는 MCP보다 가벼운 선택지
Playwright MCP브라우저 상태를 MCP 도구로 유지하며 탐색·자동화강사 시연 또는 심화. Claude Desktop/Code 등 MCP 클라이언트에 연결
Vercel · Supabase · GitHub배포, DB, 저장소·PR 작업을 에이전트가 직접 조회·관리기본 배포 후 시간이 남을 때 연결
Plugin과 MCP는 로컬 파일·브라우저·외부 계정에 접근할 수 있습니다. 수업에서는 공식 문서와 마켓플레이스 경로를 쓰고, 설치 전 출처·권한·비용을 확인합니다.
Claude Code 공식 마켓플레이스 예시
# Claude Code 안에서 실행
/plugin marketplace update claude-plugins-official
/plugin install superpowers@claude-plugins-official
/plugin install playwright@claude-plugins-official
/plugin install vercel@claude-plugins-official
/plugin install supabase@claude-plugins-official
/plugin install github@claude-plugins-official
Superpowers는 Anthropic 제작물이 아니라 공식 마켓플레이스에 등록된 서드파티 Skill입니다. 입문 수업에서는 “AI에게 일 시키는 절차가 Skill로 어떻게 바뀌는지” 보여주는 선택 예시로만 씁니다.

D. Playwright CLI/Skills — 사이트가 진짜 열리는지 에이전트가 확인

Playwright 공식 문서는 코딩 에이전트에는 CLI + Skills가 토큰 비용이 낮고 명령 흐름이 간단하다고 설명합니다. 배포 실습 뒤 URL이 뜨는지 확인할 때 좋습니다.

npm install -g @playwright/cli@latest
playwright-cli install --skills
playwright-cli open https://내-vercel-url.vercel.app --headed
playwright-cli snapshot
playwright-cli screenshot
Playwright MCP가 필요한 경우
claude mcp add playwright npx @playwright/mcp@latest
# 또는 MCP 클라이언트 JSON에 command: "npx", args: ["@playwright/mcp@latest"]
배포 URL이 열리고 첫 화면 텍스트나 버튼을 스냅샷/스크린샷으로 확인하면 성공.

E. MCP(파일시스템) 추가 — STEP 1-B와 동일

Settings > Extensions 원클릭을 기본으로 사용합니다. 수동 JSON 설정은 고급/대체 경로이며 도구 작업은 실행 전 사용자 승인을 거칩니다.

F. Cowork — 터미널 없이 에이전트로 파일 작업

  1. Claude Desktop 실행 (Cowork는 데스크탑 전용·유료)
  2. 상단 모드 선택기에서 "Cowork" 탭 클릭
  3. 할 일을 자연어로 기술 → 접근 방식 검토 → 실행
    권한: "Ask before acting"(기본·단계별 승인) / "Act without asking"(빠르지만 위험).
  4. 폴더/앱 접근이 필요하면: Settings > General의 "Computer use" 토글 켜기
    작업 시점에 앱별 접근 권한을 요청.
Cowork가 도는 동안 데스크탑 앱을 닫지 마세요 — 닫으면 세션이 종료됩니다.
내 폴더를 통째로 분석해 정리본/실파일을 만들면 성공. 확인할 것: 내 커넥터·스킬로 만든 파일

단계완료 기준
최소Vercel 라이브 URL이 열리고 템플릿 사이트가 보인다.

A. 가장 단순한 길 — 템플릿 Deploy 버튼

  1. vercel.com/templates에서 "Supabase & Next.js App Router Starter" → Deploy
  2. GitHub 로그인 → 레포가 내 GitHub로 자동 clone
    "Create Git Repository".
  3. Supabase Integration 자동 설치 → Supabase 프로젝트 생성 안내 따라가기
    완료되면 환경변수가 자동 주입되어 추가 설정 없이 동작.
  4. 배포 완료 → Vercel 라이브 URL 발급
발급된 URL을 열어 사이트가 뜨면 성공. 확인할 것: 인터넷에 떠 있는 내 사이트 URL

B. 로컬에서 시작하는 길 (대안)

npx create-next-app -e with-supabase   # 쿠키 인증+TS+Tailwind 포함
# .env.example → .env.local 로 이름 변경 후 Supabase 값 입력
npm run dev                            # http://localhost:3000
Supabase 연결에서 막히면 일반 Vercel 템플릿을 먼저 배포합니다. 수업의 최소 목표는 "인터넷에 뜨는 내 URL"입니다.

C. 내용 수정 → 자동 재배포

GitHub 레포에서 텍스트·컴포넌트를 고치고 git push하면 Vercel이 자동 재배포합니다. (클로드 코드/Antigravity로 "프로필 문구·색 바꿔줘" 위임 가능)
