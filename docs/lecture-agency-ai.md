> 발표 목표: Claude의 3단계 진화를 비개발자도 이해할 수 있게 설명하고, "AI 비서 → AI 팀"이라는 개념 전달

## AI의 3단계 진화

```
1단계: Claude (대화)         → AI 비서와 대화하기
2단계: Claude Code (코딩)    → AI 비서가 일을 실행한다
3단계: Claude Cowork (업무)  → AI 팀이 협업한다
```

## MCP — AI가 도구를 쓰는 표준

**MCP (Model Context Protocol)**: AI 에이전트가 외부 도구/서비스에 접근하는 표준 프로토콜

> Anthropic이 "AI 앱의 USB-C 포트"라고 설명

### 비개발자 비유

"지금까지 AI는 '대화만 하는 직원'이었다. MCP를 연결하면 '회사 시스템에 로그인해서 직접 일하는 직원'이 된다."

예시:
- MCP + Slack → AI가 슬랙 메시지를 읽고 답장
- MCP + Google Calendar → AI가 일정을 확인하고 회의를 잡음
- MCP + GitHub → AI가 코드를 읽고, PR을 만들고, 리뷰

### 현황

- **1,000개+** 커뮤니티 서버
- OpenAI, Google, Microsoft 모두 채택 → 사실상 유일 표준
- OpenAI가 Assistants API를 2026년 중반 종료 예정 → MCP로 통합

## 서브에이전트 — 일을 나눠주는 AI

Claude Code 안에서 특정 작업을 전담하는 **격리된 AI 인스턴스**.

```
메인 에이전트 (지시)
├── 서브에이전트 A: "이 코드 리뷰해줘"
├── 서브에이전트 B: "관련 문서 찾아줘"
└── 서브에이전트 C: "테스트 작성해줘"
```

각 서브에이전트는:
- 자기만의 컨텍스트 윈도우 (다른 에이전트와 격리)
- 전문 역할에 맞는 도구 권한
- 결과를 메인 에이전트에게 보고

기본 제공 타입:

| 타입 | 역할 | 에이전시 활용 예시 |
|------|------|-----------------|
| `general-purpose` | 범용 작업 | 클라이언트 요구사항 정리 |
| `Explore` | 코드베이스 탐색 | 기존 프로젝트 파악 |
| `Plan` | 아키텍처 설계 | 프로젝트 설계안 작성 |
| `code-reviewer` | 코드 리뷰 | QA 자동화 |

## Agent Teams — AI끼리 팀을 짠다

2026년 2월 출시. 서브에이전트의 진화형.

**핵심 차이**: 서브에이전트는 메인을 거쳐야 소통하지만, Agent Teams의 Teammate는 **서로 직접 소통** 가능

```
[서브에이전트]                  [Agent Teams]
메인 ←→ A                    Lead ←→ A ←→ B
메인 ←→ B                              ↕
(A와 B는 직접 소통 불가)             C
                            (모두 직접 소통 가능)
```

실제 사용 예시:

```
> "로그인 페이지를 리디자인해줘. 프론트, API, 테스트 동시에."

Team Lead가:
→ Frontend Agent: UI 컴포넌트 수정
→ Backend Agent: API 엔드포인트 업데이트
→ Test Agent: E2E + 단위 테스트 작성
각 에이전트가 독립적으로 작업 후 결과 통합
```

## 스킬(Skills) — AI에게 업무 매뉴얼을 가르치다

### 스킬이란?

**한 마디로: AI에게 가르쳐둔 업무 매뉴얼.**

사람으로 치면:
- 신입한테 "회의록은 이 양식으로 써" → 다음부터 알아서 함
- 인턴한테 "메일 요약은 이렇게 해" → 매번 안 알려줘도 됨

AI 스킬도 같다. SKILL.md 파일 하나 만들어두면:
1. **자동으로 인식** — "회의록 정리해줘" 하면 스킬이 알아서 발동
2. **일관된 결과** — 매번 양식 설명 안 해도 됨
3. **재사용 가능** — 모든 프로젝트에서 같은 스킬을 사용

### 두 가지 맛: Cowork 스킬 vs Claude Code 스킬

| | Cowork 스킬 | Claude Code 스킬 (SKILL.md) |
|---|---|---|
| **대상** | 비개발자 (대표/PM/기획자) | 개발자 |
| **만드는 법** | Claude Desktop에서 자연어로 작성 | `.claude/skills/` 폴더에 마크다운 파일 |
| **실행** | Claude Desktop 앱 | 터미널에서 `/스킬이름` |
| **도구** | 제한적 (대화 범위 내) | 파일 시스템, Git, API, 터미널 전체 |
| **자동화** | 스케줄 태스크 | cron, 파이프라인, 훅 연동 |
| **코딩 필요** | 없음. 한국어로 3줄이면 됨 | 마크다운 + YAML frontmatter |

**비개발자에게**: "Claude Desktop에서 3분이면 만들 수 있다. 코딩 필요 없음."

### Skills의 진화: Commands → Skills 2.0

Claude Code 초기에는 **커스텀 명령어(Commands)** 시스템이었다:

```
[Before — Commands 1.0]
.claude/commands/review.md     ← 단순 마크다운 파일
                               ← /review 로만 수동 호출
                               ← 인자 처리 없음
                               ← AI가 자동 판단 불가
```

현재는 **스킬(Skills 2.0)** 로 통합 진화:

```
[After — Skills 2.0]
.claude/skills/review/
├── SKILL.md                   ← YAML 메타데이터 + 워크플로우 정의
├── references/                ← 참고 자료 번들
├── examples/                  ← 예시 파일
└── scripts/                   ← 보조 스크립트
```

| 기능 | Commands 1.0 | Skills 2.0 |
|------|-------------|-----------|
| 호출 방식 | `/명령어`로 수동만 | 수동 `/스킬` + **AI가 맥락 보고 자동 발동** |
| 인자 전달 | 없음 | `$ARGUMENTS` 플레이스홀더 |
| 동적 컨텍스트 | 없음 | 셸 커맨드로 실시간 데이터 주입 |
| 도구 제한 | 없음 (전체 접근) | `allowed-tools`로 최소 권한 적용 |
| 실행 환경 | 메인 세션 | `context: fork`로 격리 실행 가능 |
| 서브에이전트 위임 | 불가 | `agent: Explore` 등으로 서브에이전트 실행 |
| 부작용 제어 | 없음 | `disable-model-invocation: true` |
| 파일 참조 | 없음 | `@파일경로`로 자동 로드 |
| 표준 | 독자 규격 | **Agent Skills 오픈 표준** (agentskills.io) |
| 기존 호환 | — | `.claude/commands/` 파일도 그대로 동작 |

### SKILL.md 실전 예시

```yaml
---
name: fix-issue
description: "GitHub 이슈를 분석하고 수정한다.
  '/fix-issue 1234' 형태로 호출."
disable-model-invocation: true
allowed-tools: ["Read", "Edit", "Bash(gh *)"]
---

GitHub 이슈 $ARGUMENTS를 수정한다.

1. `gh issue view`로 이슈 상세 확인
2. 코드베이스에서 관련 파일 검색
3. 수정 구현
4. 테스트 작성 및 실행
5. 커밋 + PR 생성
```

```yaml
---
name: pr-summary
description: "PR 변경 사항을 요약한다"
context: fork
agent: Explore
---

## PR 컨텍스트
- PR diff: (gh pr diff 실행)
- 변경 파일: (gh pr diff --name-only 실행)

이 PR의 변경 사항을 요약해줘.
```

### 핵심 포인트: 왜 이게 에이전시에 중요한가?

1. **반복 업무의 표준화**: 프로젝트마다 같은 QA 체크, 코드 리뷰, 리포트 → 스킬로 한 번 만들면 끝
2. **팀원 온보딩 비용 절감**: 새 개발자가 `/fix-issue`, `/code-review` 스킬만 알면 바로 생산성 발휘
3. **품질 일관성**: 사람마다 다른 리뷰 품질 → 스킬이 기준선을 보장
4. **오픈 표준**: Agent Skills 표준을 따르므로 다른 AI 도구와도 호환 가능

## Agent SDK — 프로그래밍적 제어 (개발자 대상)

Agent SDK는 Claude Code의 기능을 Python/TypeScript로 직접 제어.

**에이전시에 의미하는 것**:
- 반복되는 프로젝트 셋업을 코드로 자동화
- 에이전트 파이프라인을 프로그래밍적으로 구성
- 클라이언트별 맞춤 워크플로우 구축

```python
# 예: QA 전용 에이전트 (Agent SDK)
agent = ClaudeAgent(
    model="claude-sonnet-4-6",
    tools=[code_search, test_runner],
    system_prompt="변경된 코드의 테스트를 작성하고 실행해."
)
```

# Module 3: 에이전시에 에이전틱 팀 도입하기 (25분) ★메인

> 발표 목표: 에이전시 업무 흐름의 각 단계에서 AI가 어떻게 활용되는지 구체적으로 보여주고, 실제 운영 사례와 비용을 현실적으로 전달

## 에이전시 업무 흐름 × AI 자동화 매핑

에이전시의 전형적인 프로젝트 흐름과 AI 자동화 포인트:

| 단계 | 전통적 방식 | AI 에이전트 활용 | 자동화 수준 |
|------|-----------|----------------|-----------|
| **리서치** | 사람이 구글링 + 정리 | 리서치 에이전트가 자료 수집 → 구조화된 보고서 | 매우 높음 |
| **기획** | PM이 문서 작성 | AI가 리서치 기반으로 기획서 초안 작성 | 높음 |
| **디자인** | 디자이너 목업 | AI가 와이어프레임 + UI 초안 생성 | 보통 |
| **개발** | 개발자 코딩 | 코딩 에이전트가 구현 + 개발자가 리뷰 | 높음 |
| **QA** | QA 수동 테스트 | QA 에이전트가 테스트 코드 작성 + 실행 | 매우 높음 |
| **납품/운영** | 수동 배포 + 문서화 | 자동 배포 파이프라인 + 문서 자동 생성 | 높음 |

**핵심 메시지**: "모든 단계를 AI로 대체하는 게 아니다. **반복적이고 구조화된 작업을 AI에게 맡기고, 사람은 판단과 창의에 집중**하는 것이다."

## 실제 사례 — codemon.ai의 AI 팀

codemon 팀은 7마리의 AI 에이전트로 구성:

| 에이전트 | 역할 | 담당 업무 |
|---------|------|----------|
| 로디 | PM & 오케스트레이터 | 기획, 블로그 작성, 팀 조율 |
| 누비 | QA & Dev | 코드 리뷰, 테스트, 디버깅, 외주 대응 |
| 나래 | Content & Marketing | 리서치, 자료 수집, SEO |
| 베어 | Design & Frontend | UI/UX, 컴포넌트 설계 |
| 뉴비 | Research & Scout | 트렌드 분석, 경쟁사 조사 |
| 옥토 | Backend & Infra | 서버, API, 인프라 |
| 아울 | Data & Automation | 데이터 처리, 자동화 |

**에이전시 관점 포인트**: "1인 개발자가 7명의 AI 팀을 운영한다. 에이전시가 이 구조를 도입하면?"

## CLAUDE.md = AI 팀원의 온보딩 문서

새 팀원이 입사하면 온보딩 문서를 준다. AI 에이전트도 마찬가지.

**CLAUDE.md**: 프로젝트에 들어올 때 AI가 가장 먼저 읽는 파일

```markdown
# CLAUDE.md (예시)

## 프로젝트 개요
- 클라이언트: ○○ 에이전시
- 스택: Next.js 15 + Supabase + Vercel
- 배포: vercel build && vercel deploy --prebuilt

## 코딩 규칙
- 커밋 메시지: 한국어, 상세하게
- 테스트: 새 기능마다 반드시 작성
- 금지: rm -rf, force push to main

## 에이전트 역할
- code-reviewer: 모든 PR에 대해 보안/성능 리뷰
- research-scout: 기술 선택 시 대안 조사
```

**비개발자에게**: "이 문서 하나만 잘 쓰면, AI가 우리 팀의 규칙을 이해하고 따른다. 회사 컨벤션, 금지 사항, 선호하는 방식을 다 적어두면 된다."

## 훅(Hooks) = 자동 가드레일

에이전트가 자율적으로 일할 때 필요한 안전장치.

> "훅은 AI한테 거는 제한이 아니라, 업무 프로세스를 자동화하는 것이다."

| 훅 | 시점 | 에이전시 활용 예시 |
|----|------|-----------------|
| `PreToolUse` | 도구 실행 전 | 프로덕션 DB 직접 수정 차단 |
| `PostToolUse` | 도구 실행 후 | 코드 수정 시 자동 포맷팅 + 린트 |
| `SessionStart` | 세션 시작 | 프로젝트별 컨텍스트 자동 로드 |
| `Stop` | 작업 완료 | 슬랙/텔레그램으로 결과 알림 |

**실제 예시**: "개발 에이전트가 코드를 수정하면 → 자동으로 prettier가 돌고 → 테스트가 실행되고 → 실패하면 에이전트가 다시 수정 → 성공하면 슬랙에 알림"

## 스킬(Skills) = AI용 업무 매뉴얼

스킬은 재사용 가능한 워크플로우를 정의한 문서.

```markdown
# 리서치 스킬 (SKILL.md)
1. 주제 키워드로 웹 검색 (최소 5개 소스)
2. 각 소스에서 핵심 정보 추출
3. 결과를 구조화된 마크다운으로 정리
4. 출처 목록 포함
```

**에이전시 업무별 스킬 아이디어**:

| 스킬 | 용도 | 담당 에이전트 |
|------|------|-------------|
| 경쟁사 분석 | 클라이언트 경쟁사 5곳 조사 → 보고서 | 뉴비 (Research) |
| 기술 스택 추천 | 요구사항 기반 기술 스택 비교표 | 옥토 (Backend) |
| 코드 리뷰 | PR별 보안/성능/스타일 검토 | 누비 (QA) |
| 주간 보고서 | 프로젝트 진행 상황 자동 요약 | 로디 (PM) |
| 클라이언트 QA | 체크리스트 기반 기능 검증 | 아울 (Data) |

**비개발자에게**: "업무 매뉴얼을 문서로 작성하듯, AI한테 일 시키는 매뉴얼을 작성하는 것이다. 한 번 만들면 모든 프로젝트에서 재사용된다."

## 멀티 에이전트 파이프라인 — 실전 워크플로우

codemon 팀에서 블로그 글 하나를 만드는 파이프라인:

```
1. 로디(PM) → 주제 선정, 키워드 기획     [1분]
2. 나래(리서치) → 자료 수집, 보고서 작성   [5분]
3. 로디(블로그) → 블로그 글 작성          [2분]
4. 누비(QA) → 팩트체크, 검수, 발행        [5분]
────────────────────────────────────────
총 13분 → 발행 가능한 블로그 글 완성
```

**v1(36분) → v2(13분) → v3(9분)**: 파이프라인 최적화로 75% 시간 단축

**에이전시 프로젝트에 적용한다면**:

```
1. 뉴비(리서치) → 클라이언트 산업/경쟁사 조사   [10분]
2. 로디(기획) → 프로젝트 기획서 초안             [5분]
3. 베어(디자인) → 와이어프레임 초안               [15분]
4. 옥토(개발) → 프론트엔드 스캐폴딩              [20분]
5. 누비(QA) → 코드 리뷰 + 기본 테스트           [10분]
────────────────────────────────────────────
총 ~1시간 → 클라이언트 프레젠테이션용 프로토타입
```

## 2봇 협업 — 현실적인 시작점

모든 에이전시가 7마리 에이전트부터 시작할 필요 없다. **2대부터 시작하자.**

codemon의 시작: 맥미니 1대(로디) + 맥북 1대(누비)

**협업 구조**:
- iCloud 공유 폴더로 비동기 통신
- GitHub으로 스킬 공유
- 역할 비대칭: 로디=상주 서버 관리자, 누비=이동형 보조

**핵심 설계 원칙**:

| 원칙 | 이유 |
|------|------|
| 비동기 > 실시간 | 상시 연결 불필요한 환경에서 과잉 설계 방지 |
| 파일 > 프로토콜 | 디버깅 쉽고, 사람도 읽을 수 있고, 도구 의존성 없음 |
| 역할 분리 | 대칭 구조는 충돌을 유발. 비대칭이 자연스럽다 |

# Module 4: 라이브 데모 (15분)

> 발표 목표: "이론이 아니라 진짜 동작하는 시스템"을 보여줘서 신뢰 확보

## 데모 시나리오 A: 텔레그램에서 블로그 만들기 (메인)

### 스토리

"제가 지금 텔레그램에서 한 줄 메시지를 보내겠습니다. 그러면 AI 팀 3마리가 자동으로 움직여서 블로그 글을 만듭니다."

### 실행 순서

1. **텔레그램에 메시지 전송** (라이브)
   ```
   "에이전시에서 AI 자동화 도입하는 방법에 대해 글 써줘"
   ```

2. **라이브 프로그레스 표시** (텔레그램 메시지 1개가 실시간 업데이트)
   ```
   에이전시 AI 자동화
   ▓▓▓▓▓▓░░░░░░░░░░░░░░ 1/3
   나래 리서치 진행 중... (2분)
   ```

3. **각 단계 설명** (프로그레스가 진행되는 동안)
   - Step 1: 나래가 웹에서 자료 수집 (~3분)
   - Step 2: 로디가 리서치 결과를 기반으로 글 작성 (~2분)
   - Step 3: 누비가 팩트체크 + 검수 (~4분)

4. **완료 메시지**
   ```
   에이전시 AI 자동화 ✅ 완료!
   ▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓ 3/3
   ✅ 나래 리서치 (3분)
   ✅ 로디 블로그 (2분)
   ✅ 누비 검수 (4분)
   ```

5. **결과물 보여주기**: 옵시디언에서 생성된 블로그 글 열기

### 데모 중 보여줄 포인트

| 포인트 | 설명 | 청중 반응 유도 |
|--------|------|--------------|
| 메시지 1개로 시작 | 복잡한 설정 없이 한 줄로 | "이렇게 간단합니다" |
| 라이브 프로그레스 | 진행 상황이 실시간으로 보임 | "어디까지 됐는지 볼 수 있습니다" |
| 에이전트 역할 분담 | 리서치→작성→검수 | "사람 팀처럼 역할이 나뉘어 있습니다" |
| 9~13분에 완성 | 실제 시간 체험 | "이 시간 동안 다른 일을 할 수 있습니다" |
| 결과물 품질 | 발행 가능한 수준 | "검수까지 자동으로 됩니다" |

## 데모 시나리오 B: Claude Code 멀티 에이전트 (백업/추가)

### Claude Code에서 직접 보여주기

```bash
$ claude
> 이 프로젝트의 로그인 페이지 코드를 리뷰하고,
  보안 이슈가 있으면 수정해줘.
  서브에이전트로 리뷰와 수정을 병렬로 진행해.
```

보여줄 것:
- 서브에이전트가 자동으로 생성되는 모습
- 코드 리뷰 결과
- 자동 수정 제안

## 데모 시나리오 C: 상태 조회 + 중단 (인터랙션)

파이프라인이 돌아가는 동안:

```
나: "상태" (텔레그램)
봇: (1초 이내 즉시 응답) 현재 진행 상황 표시

나: "중단해"
봇: 파이프라인 중단 + 완료된 산출물 보존 알림
```

"AI한테 일을 시켰는데 중간에 방향을 바꾸고 싶을 때, 즉시 멈출 수 있습니다."

## 기술 준비사항

### 사전 준비

- 인터넷 연결 확인 (회장 Wi-Fi 테스트)
- 텔레그램 봇 정상 동작 확인

### Lv.1 — 오늘 할 수 있는 것 (0원, 30분)

| 액션 | 방법 | 대상 |
|------|------|------|
| **Claude.ai 가입** | claude.ai 접속, 무료 플랜 시작 | 전체 |
| **업무 프롬프트 만들기** | 반복하는 업무를 Claude에게 시켜보기 | 전체 |
| **회의록 요약** | 회의 내용 붙여넣기 → 요약 + 액션 아이템 추출 | PM/대표 |
| **코드 리뷰 요청** | 코드 붙여넣기 → "이 코드 리뷰해줘" | 개발자 |

### Lv.2 — 이번 주에 할 수 있는 것 ($20/월, 2시간)

| 액션 | 방법 | 대상 |
|------|------|------|
| **Claude Pro 가입** | 월 $20 → 더 많은 대화, Claude Cowork 접근 | 전체 |

| **CLAUDE.md 작성** | 프로젝트 루트에 팀 규칙/컨벤션 문서화 | 개발자 |
| **첫 스킬 만들기** | Cowork에서 반복 업무를 스킬로 정의 | PM/대표 |

```bash
# Claude Code 설치 (개발자용)

```

## Q&A

### 예상 질문과 답변

**Q: "코드 모르는 사람도 할 수 있나요?"**
A: Lv.1은 누구나 가능합니다. Claude.ai에서 대화하고, Cowork에서 스킬을 만드는 건 코딩 경험 없이도 됩니다. Lv.2부터는 개발자의 도움이 필요합니다.

**Q: "보안은 괜찮나요? 우리 코드가 학습에 쓰이나요?"**
A: Claude는 API/Pro 플랜에서 사용자 데이터로 모델을 학습하지 않습니다. 엔터프라이즈 플랜은 추가 보안 보장이 있습니다.

**Q: "비용이 계속 늘어나지 않나요?"**
A: 모델 분리(Opus/Sonnet/Haiku)와 프롬프트 캐싱으로 최적화하면 개발자 1인 월 $100~200 수준으로 유지 가능합니다. ROI는 보통 첫 달부터 체감됩니다.

**Q: "우리 에이전시 규모(5~10명)에도 효과가 있나요?"**
A: 오히려 소규모 에이전시에 더 효과적입니다. 대기업은 도입 프로세스가 복잡하지만, 소규모 팀은 이번 주부터 시작할 수 있습니다.

**Q: "어떤 업무부터 자동화하면 좋나요?"**
A: 반복적이고, 구조화되어 있고, 실패해도 치명적이지 않은 업무부터. 추천: 경쟁사 리서치, 코드 리뷰, 주간 보고서 자동 생성.
